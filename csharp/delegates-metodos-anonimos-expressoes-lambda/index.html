Delegates, Métodos Anônimos e Expressões Lambda

Delegates, métodos anônimos e expressões lambda​

Até agora, já utilizamos muitos métodos.

Também já passamos valores para métodos usando parâmetros.

Por exemplo:

public void ExibirMensagem(string mensagem)
{
    Console.WriteLine(mensagem);
}

Nesse caso, o método recebe um texto.

Mas, em algumas situações, queremos passar um comportamento como parâmetro.

Ou seja, queremos que um método receba outro método.

Para isso, o C# possui os delegates.

O problema que os delegates resolvem​

Imagine que temos uma lista de produtos.

Queremos filtrar apenas os produtos que custam mais de 100.

Poderíamos escrever um método específico para isso:

public static bool PrecoMaiorQueCem(Produto produto)
{
    return produto.Preco > 100;
}

Depois, poderíamos usar esse método em alguma lógica de filtro.

Mas e se em outra tela precisarmos filtrar:

  • produtos com estoque baixo;
  • produtos de uma categoria específica;
  • produtos ativos;
  • produtos com nome começando com uma letra.

Criar um método de filtro diferente para cada caso pode deixar o código repetitivo.

Seria melhor se o filtro pudesse receber uma regra.

Essa regra é um comportamento.

E delegates permitem representar esse comportamento em uma variável ou parâmetro.

O que é um delegate?​

Um delegate é um tipo que representa a assinatura de um método.

Ele define:

  • o tipo de retorno;
  • os parâmetros que o método deve receber.

Exemplo:

public delegate bool RegraDeFiltro(Produto produto);

Esse delegate representa qualquer método que:

  • receba um Produto;
  • retorne um bool.

Então este método combina com o delegate:

public static bool ProdutoCaro(Produto produto)
{
    return produto.Preco > 100;
}

Ele combina porque recebe um Produto e retorna bool.

Um delegate permite guardar uma referência para um método compatível com sua assinatura.

Usando um delegate​

Imagine a classe Produto:

public class Produto
{
    public string Nome { get; set; }
    public decimal Preco { get; set; }
    public int Estoque { get; set; }
}

Agora podemos criar uma variável do tipo RegraDeFiltro:

RegraDeFiltro regra = ProdutoCaro;

Depois, podemos chamar essa regra como se fosse um método:

Produto produto = new Produto
{
    Nome = "Teclado",
    Preco = 150,
    Estoque = 8
};

bool resultado = regra(produto);

Console.WriteLine(resultado);

Nesse exemplo, resultado será true.

A variável regra guarda uma referência para o método ProdutoCaro.

Passando um método como parâmetro​

Agora podemos criar um método que recebe uma regra de filtro:

public static List<Produto> FiltrarProdutos(
    List<Produto> produtos,
    RegraDeFiltro regra
)
{
    List<Produto> produtosFiltrados = new List<Produto>();

    foreach (Produto produto in produtos)
    {
        if (regra(produto))
            produtosFiltrados.Add(produto);
    }

    return produtosFiltrados;
}

Observe esta linha:

if (regra(produto))

O método FiltrarProdutos não sabe qual regra será usada.

Ele apenas sabe que receberá uma regra compatível com o delegate.

Isso deixa o método mais flexível.

Exemplo com métodos nomeados​

Podemos criar regras diferentes:

public static bool ProdutoCaro(Produto produto)
{
    return produto.Preco > 100;
}

public static bool ProdutoComEstoqueBaixo(Produto produto)
{
    return produto.Estoque < 5;
}

Depois, podemos escolher qual regra passar:

List<Produto> produtosCaros = FiltrarProdutos(produtos, ProdutoCaro);

List<Produto> produtosComEstoqueBaixo = FiltrarProdutos(
    produtos,
    ProdutoComEstoqueBaixo
);

O mesmo método FiltrarProdutos funciona para regras diferentes.

Quem muda é o comportamento enviado por parâmetro.

Métodos anônimos​

Às vezes, a regra é pequena e será usada apenas uma vez.

Nesses casos, criar um método nomeado pode ser exagerado.

O C# permite criar um método sem nome usando delegate.

Esse recurso é chamado de método anônimo.

Exemplo:

List<Produto> produtosCaros = FiltrarProdutos(
    produtos,
    delegate (Produto produto)
    {
        return produto.Preco > 100;
    }
);

Nesse exemplo, não criamos um método chamado ProdutoCaro.

Criamos a regra diretamente no local onde ela será usada.

Esse formato funciona, mas ainda é um pouco verboso.

Por isso, no dia a dia, usamos mais expressões lambda.

Expressões lambda​

Uma expressão lambda é uma forma curta de escrever um comportamento.

Ela usa o operador =>.

Podemos ler esse operador como:

"vai para" ou "resulta em".

O exemplo com método anônimo pode ser escrito assim:

List<Produto> produtosCaros = FiltrarProdutos(
    produtos,
    produto => produto.Preco > 100
);

Esse código quer dizer:

dado um produto, retorne se produto.Preco é maior que 100.

A lambda é mais curta, mas representa a mesma ideia:

  • recebe um Produto;
  • retorna um bool;
  • pode ser usada onde um delegate compatível é esperado.

Lambda com mais de uma linha​

Quando a lógica é maior, podemos usar chaves:

List<Produto> produtosValidos = FiltrarProdutos(
    produtos,
    produto =>
    {
        bool temNome = !string.IsNullOrWhiteSpace(produto.Nome);
        bool temPrecoValido = produto.Preco > 0;

        return temNome && temPrecoValido;
    }
);

Nesse formato, precisamos usar return.

Ele é útil quando a regra possui mais de uma instrução.

Func e Action​

Na prática, nem sempre precisamos declarar nossos próprios delegates.

O .NET já possui delegates genéricos muito usados.

Dois dos principais são:

  • Func, usado quando existe retorno;
  • Action, usado quando não existe retorno.

Exemplo com Func:

Func<Produto, bool> regra = produto => produto.Preco > 100;

bool resultado = regra(produto);

O último tipo informado em Func é o retorno.

Neste caso:

  • Produto é o parâmetro;
  • bool é o retorno.

Exemplo com Action:

Action<Produto> exibirProduto = produto =>
{
    Console.WriteLine(produto.Nome);
};

exibirProduto(produto);

Nesse caso, a ação recebe um Produto, mas não retorna valor.

Predicate​

Também existe o Predicate<T>.

Ele representa uma regra que recebe um valor e retorna bool.

Exemplo:

Predicate<Produto> estaEmPromocao = produto => produto.Preco < 50;

bool resultado = estaEmPromocao(produto);

Esse tipo é comum quando queremos representar uma pergunta.

Por exemplo:

  • este produto é caro?
  • este usuário está ativo?
  • esta data já passou?
  • este item deve aparecer na tela?

Como isso aparece no LINQ​

Delegates e lambdas aparecem todos os dias no .NET.

Um dos lugares mais comuns é o LINQ.

LINQ é um conjunto de métodos usado para consultar e transformar coleções.

Exemplo:

List<Produto> produtosCaros = produtos
    .Where(produto => produto.Preco > 100)
    .ToList();

O método Where recebe uma regra.

Essa regra decide se cada item deve permanecer na lista.

Neste caso:

produto => produto.Preco > 100

é uma lambda.

Ela será executada para cada produto da coleção.

Outros exemplos com LINQ​

Podemos ordenar produtos:

List<Produto> produtosOrdenados = produtos
    .OrderBy(produto => produto.Nome)
    .ToList();

Podemos transformar produtos em textos:

List<string> nomes = produtos
    .Select(produto => produto.Nome)
    .ToList();

Podemos verificar se existe algum produto com estoque baixo:

bool existeEstoqueBaixo = produtos
    .Any(produto => produto.Estoque < 5);

Podemos contar produtos ativos em uma determinada condição:

int quantidadeProdutosCaros = produtos
    .Count(produto => produto.Preco > 100);

Em todos esses exemplos, estamos passando comportamentos para métodos do LINQ.

Esse comportamento normalmente é escrito com lambda.

Uso diário em aplicações ASP.NET MVC​

Em aplicações ASP.NET MVC, lambdas aparecem em vários lugares.

Um exemplo comum é transformar Models em ViewModels:

List<ListarFabricanteViewModel> visualizarFabricantes = fabricantes
    .Select(fabricante => new ListarFabricanteViewModel(
        fabricante.Id,
        fabricante.Nome,
        fabricante.Email,
        fabricante.Telefone
    ))
    .ToList();

Nesse caso, o Select recebe uma lambda.

Ela ensina o LINQ a transformar cada Fabricante em uma ListarFabricanteViewModel.

Também podemos procurar um registro:

Fabricante? fabricante = fabricantes
    .FirstOrDefault(fabricante => fabricante.Id == id);

Aqui, a lambda representa a regra de busca.

O método FirstOrDefault percorre a coleção até encontrar o primeiro item que atende essa regra.

Uso diário com eventos​

Delegates também são a base dos eventos no .NET.

Um evento representa algo que aconteceu.

Por exemplo:

  • um botão foi clicado;
  • uma requisição foi recebida;
  • um item foi selecionado;
  • uma tarefa foi concluída.

Em aplicações com interface gráfica, é comum escrever algo assim:

botaoSalvar.Click += (sender, args) =>
{
    SalvarRegistro();
};

Nesse exemplo, a lambda descreve o que deve acontecer quando o botão for clicado.

Mesmo que em ASP.NET MVC tradicional não trabalhemos com clique de botão dessa forma no servidor, a ideia é importante.

Muitos recursos do .NET usam delegates para permitir que nosso código informe o comportamento desejado.

Por que isso é importante?​

Delegates, métodos anônimos e lambdas permitem escrever código mais flexível.

Com eles, conseguimos:

  • passar regras como parâmetro;
  • evitar repetição de código;
  • criar filtros e transformações de forma clara;
  • usar LINQ com naturalidade;
  • entender melhor APIs modernas do .NET.

No começo, a sintaxe pode parecer diferente.

Mas a ideia central é simples:

em vez de passar apenas dados, também podemos passar comportamentos.

Comparando as três formas​

As três formas abaixo representam a mesma regra.

Com método nomeado:

public static bool ProdutoCaro(Produto produto)
{
    return produto.Preco > 100;
}

List<Produto> produtosCaros = FiltrarProdutos(produtos, ProdutoCaro);

Com método anônimo:

List<Produto> produtosCaros = FiltrarProdutos(
    produtos,
    delegate (Produto produto)
    {
        return produto.Preco > 100;
    }
);

Com expressão lambda:

List<Produto> produtosCaros = FiltrarProdutos(
    produtos,
    produto => produto.Preco > 100
);

No dia a dia, a lambda costuma ser a opção mais usada.

Ela é curta e combina muito bem com LINQ.

Mas entender delegates ajuda a perceber o que está acontecendo por trás da sintaxe.

Conclusão​

Um delegate representa um método compatível com uma assinatura.

Métodos anônimos permitem criar esse comportamento sem declarar um método separado.

Expressões lambda deixam essa escrita mais curta e mais comum no C# moderno.

Esses conceitos aparecem diariamente no .NET, principalmente em:

  • consultas com LINQ;
  • transformações com Select;
  • filtros com Where;
  • buscas com FirstOrDefault;
  • verificações com Any;
  • eventos;
  • callbacks e configurações de bibliotecas.

Ao entender lambdas, o código com LINQ deixa de parecer mágico.

Ele passa a ser apenas uma forma elegante de passar comportamento para métodos.
