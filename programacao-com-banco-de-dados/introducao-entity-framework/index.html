Introdução ao Entity Framework

Introdução ao Entity Framework​

Nas aulas anteriores, vimos duas formas importantes de pensar o banco de dados.

Primeiro, vimos o SQL Server como o lugar onde os dados ficam armazenados.

Depois, vimos o Dapper como uma biblioteca que ajuda o C# a executar comandos SQL.

Agora vamos conhecer outra ferramenta muito usada em aplicações .NET:

o Entity Framework Core, também chamado de EF Core.

O Entity Framework é uma biblioteca que permite trabalhar com o banco de dados usando objetos C#.

Ele não elimina o banco.

Ele também não elimina a importância de entender SQL.

Mas ele reduz bastante o código manual necessário para buscar, cadastrar, editar e excluir registros.

O problema que o Entity Framework resolve​

Quando usamos SQL diretamente, precisamos escrever comandos como:

  • SELECT;
  • INSERT;
  • UPDATE;
  • DELETE;
  • JOIN.

Também precisamos abrir conexão, passar parâmetros e transformar linhas do banco em objetos C#.

Com Dapper, parte desse trabalho fica mais simples.

Mas ainda escrevemos bastante SQL manualmente.

Com Entity Framework, a ideia muda um pouco.

Em vez de escrever SQL para cada operação simples, usamos classes, propriedades e métodos do C#.

O EF Core traduz muitas dessas operações para SQL.

O que é um ORM?​

Entity Framework Core é um ORM.

ORM significa Object-Relational Mapper.

Em português, podemos entender como:

uma ferramenta que mapeia objetos C# para tabelas de um banco relacional.

Essa ponte é importante porque a aplicação trabalha com objetos.

O banco trabalha com tabelas.

O ORM fica entre esses dois mundos.

Code First​

Existem formas diferentes de usar o Entity Framework.

Nesta aula, vamos usar a abordagem Code First.

Code First significa:

o código C# descreve o modelo da aplicação, e o Entity Framework usa esse modelo para criar ou atualizar o banco.

Nessa abordagem, começamos criando as classes do projeto.

Depois, configuramos como essas classes serão transformadas em tabelas.

Por fim, usamos migrations para gerar as mudanças no banco.

Essa abordagem combina bem com projetos em que:

  • a aplicação está sendo construída junto com o banco;
  • queremos versionar mudanças de estrutura pelo código;
  • queremos revisar alterações de banco no Git;
  • queremos evitar criar tabelas manualmente no SQL Server.

Entidades​

No Entity Framework, uma classe que representa um dado importante do sistema costuma ser chamada de entidade.

No sistema Controle de Medicamentos, podemos ter uma entidade para fornecedor:

public class Fornecedor
{
    public Guid Id { get; set; }

    public string Nome { get; set; } = string.Empty;

    public string Telefone { get; set; } = string.Empty;

    public string Cnpj { get; set; } = string.Empty;
}

Essa classe ainda não é uma tabela.

Ela é uma classe C#.

Mas o Entity Framework consegue usar essa classe como base para criar uma tabela no banco.

Em uma ideia simples:

  • a classe pode virar uma tabela;
  • as propriedades podem virar colunas;
  • o Id pode virar chave primária.

O DbContext​

O DbContext é uma das classes mais importantes do Entity Framework.

Ele representa uma sessão de trabalho com o banco de dados.

Em linguagem simples:

o DbContext é o objeto que a aplicação usa para consultar e salvar dados com EF Core.

Um contexto inicial poderia ser assim:

using Microsoft.EntityFrameworkCore;

public class ControleDeMedicamentosDbContext : DbContext
{
    public ControleDeMedicamentosDbContext(DbContextOptions<ControleDeMedicamentosDbContext> options)
        : base(options)
    {
    }

    public DbSet<Fornecedor> Fornecedores { get; set; }
}

O DbSet<Fornecedor> representa o conjunto de fornecedores que será controlado pelo EF Core.

Na prática, ele indica que queremos trabalhar com fornecedores no banco.

Por que configurar as entidades?​

O Entity Framework consegue descobrir algumas coisas sozinho.

Por exemplo, uma propriedade chamada Id normalmente é entendida como chave primária.

Mas nem tudo deve ficar implícito.

Em um projeto real, precisamos controlar detalhes como:

  • nome da tabela;
  • chave primária;
  • tamanho máximo de textos;
  • campos obrigatórios;
  • relacionamentos;
  • nomes de colunas;
  • índices.

Poderíamos colocar essas configurações diretamente no DbContext.

Mas, conforme o projeto cresce, o DbContext ficaria grande demais.

Por isso, uma forma organizada é usar IEntityTypeConfiguration.

O que é IEntityTypeConfiguration?​

IEntityTypeConfiguration é uma interface do Entity Framework.

Ela permite criar uma classe de configuração para uma entidade específica.

Em vez de colocar todas as configurações dentro do DbContext, criamos um arquivo separado para cada entidade.

Exemplo:

using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class FornecedorConfiguration : IEntityTypeConfiguration<Fornecedor>
{
    public void Configure(EntityTypeBuilder<Fornecedor> builder)
    {
        builder.ToTable("TBFornecedor");

        builder.HasKey(fornecedor => fornecedor.Id);

        builder.Property(fornecedor => fornecedor.Nome)
            .HasMaxLength(100)
            .IsRequired();

        builder.Property(fornecedor => fornecedor.Telefone)
            .HasMaxLength(15)
            .IsRequired();

        builder.Property(fornecedor => fornecedor.Cnpj)
            .HasMaxLength(18)
            .IsRequired();
    }
}

Essa classe explica ao EF Core como a entidade Fornecedor deve ser mapeada.

Vamos ler o código por partes.

builder.ToTable("TBFornecedor");

Define o nome da tabela no banco.

builder.HasKey(fornecedor => fornecedor.Id);

Define a chave primária.

builder.Property(fornecedor => fornecedor.Nome)
    .HasMaxLength(100)
    .IsRequired();

Define que Nome será obrigatório e terá tamanho máximo de 100 caracteres.

Aplicando as configurações no DbContext​

Depois de criar a configuração, precisamos avisar o DbContext.

Uma forma direta seria aplicar uma configuração por vez:

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfiguration(new FornecedorConfiguration());
}

Mas, em projetos com várias entidades, existe uma forma mais prática.

Podemos pedir para o EF Core procurar as configurações no assembly do projeto:

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfigurationsFromAssembly(typeof(ControleDeMedicamentosDbContext).Assembly);
}

Com isso, o EF Core procura classes que implementam IEntityTypeConfiguration.

Depois, aplica essas configurações automaticamente.

O contexto ficaria assim:

using Microsoft.EntityFrameworkCore;

public class ControleDeMedicamentosDbContext : DbContext
{
    public ControleDeMedicamentosDbContext(DbContextOptions<ControleDeMedicamentosDbContext> options)
        : base(options)
    {
    }

    public DbSet<Fornecedor> Fornecedores { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(ControleDeMedicamentosDbContext).Assembly);
    }
}

Assim, o DbContext continua mais limpo.

Cada entidade fica com sua própria configuração.

Requisitos antes da instalação​

Antes de integrar o Entity Framework ao projeto, precisamos ter alguns requisitos.

O primeiro requisito é o SDK do .NET instalado.

Podemos verificar com:

dotnet --version

O segundo requisito é um projeto .NET criado.

Por exemplo, uma aplicação ASP.NET MVC.

O terceiro requisito é um SQL Server disponível.

No Code First, o banco pode ser criado ou atualizado pelas migrations.

O quarto requisito é uma connection string.

Ela informa ao projeto onde o banco será criado ou acessado.

Connection string​

A connection string normalmente fica no arquivo appsettings.json.

Exemplo:

{
  "ConnectionStrings": {
    "ControleDeMedicamentosWeb": "Server=(localdb)\\MSSQLLocalDB;Database=ControleDeMedicamentosWeb;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}

Essa configuração diz:

  • qual servidor será usado;
  • qual banco será acessado;
  • como a autenticação será feita;
  • se o certificado do servidor será confiável no ambiente local.

Atenção: connection strings podem conter informações sensíveis. Em projetos reais, evite colocar usuário e senha diretamente no código-fonte.

Instalando a ferramenta do Entity Framework​

Para executar comandos do Entity Framework pelo terminal, usamos a ferramenta dotnet ef.

Se ela ainda não estiver instalada, use:

dotnet tool install --global dotnet-ef

Se ela já estiver instalada e você quiser atualizar, use:

dotnet tool update --global dotnet-ef

Depois, podemos verificar se o comando está disponível:

dotnet ef --version

Essa ferramenta será usada para criar e aplicar migrations.

Instalando os pacotes no projeto​

Dentro da pasta do projeto ASP.NET MVC, precisamos instalar os pacotes do EF Core.

Para usar SQL Server, instalamos:

dotnet add package Microsoft.EntityFrameworkCore.SqlServer

Esse pacote ensina o EF Core a conversar com SQL Server.

Também instalamos o pacote de design:

dotnet add package Microsoft.EntityFrameworkCore.Design

Esse pacote é usado pelas ferramentas do Entity Framework durante comandos como criação de migrations e atualização do banco.

Registrando o DbContext no ASP.NET​

Depois de criar o contexto, precisamos registrá-lo na aplicação.

Em uma aplicação ASP.NET Core, isso normalmente fica no Program.cs.

Exemplo:

using Microsoft.EntityFrameworkCore;

builder.Services.AddDbContext<ControleDeMedicamentosDbContext>(options =>
{
    string connectionString = builder.Configuration.GetConnectionString("ControleDeMedicamentosWeb")!;

    options.UseSqlServer(connectionString);
});

Esse registro diz ao ASP.NET:

quando alguém precisar de ControleDeMedicamentosDbContext, crie um contexto configurado para SQL Server.

Criando a primeira migration​

Depois de criar as entidades, configurar o DbContext e registrar a connection string, podemos criar uma migration.

Uma migration representa uma mudança na estrutura do banco.

No primeiro momento, ela representa a criação inicial das tabelas.

O comando é:

dotnet ef migrations add InitialCreate

Esse comando gera uma pasta de migrations no projeto.

Dentro dela, o EF Core cria arquivos C# descrevendo o que deve acontecer no banco.

Por exemplo:

  • criar a tabela TBFornecedor;
  • criar a coluna Nome;
  • definir o tamanho máximo da coluna;
  • marcar campos obrigatórios;
  • criar a chave primária.

Aplicando a migration no banco​

Criar a migration não altera o banco imediatamente.

Ela apenas gera o código da mudança.

Para aplicar a migration no SQL Server, usamos:

dotnet ef database update

Esse comando executa as migrations pendentes.

Se o banco ainda não existir, o EF Core pode criá-lo.

Se o banco já existir, ele aplica as mudanças necessárias.

Usando o contexto em uma classe​

Depois do registro, uma classe pode receber o contexto por injeção de dependência.

Exemplo:

public class RepositorioFornecedorEf(ControleDeMedicamentosDbContext dbContext)
{
    public List<Fornecedor> SelecionarTodos()
    {
        return dbContext.Fornecedores
            .OrderBy(fornecedor => fornecedor.Nome)
            .ToList();
    }
}

Nesse exemplo, não escrevemos um SELECT manualmente.

Escrevemos uma consulta em C#.

O Entity Framework transforma essa consulta em SQL.

Diferença entre Dapper e Entity Framework​

Dapper e Entity Framework resolvem problemas parecidos, mas com estilos diferentes.

Com Dapper, o SQL fica explícito no código.

Exemplo:

return conexao.Query<Fornecedor>("SELECT Id, Nome, Telefone, Cnpj FROM dbo.TBFornecedor").ToList();

Com Entity Framework, usamos objetos e consultas C#.

Exemplo:

return dbContext.Fornecedores.ToList();

Nenhuma das duas ferramentas é automaticamente melhor em todos os casos.

Elas apenas fazem escolhas diferentes.

Nesta semana, o foco será entender o Entity Framework.

Cuidados no Code First​

Ao usar Code First, o código passa a ser a principal descrição da estrutura do banco.

Isso exige cuidado.

Se uma propriedade for removida de uma entidade, uma migration pode tentar remover uma coluna do banco.

Se o tamanho de uma propriedade mudar, uma migration pode alterar o tipo da coluna.

Se um relacionamento mudar, a estrutura do banco também pode mudar.

Por isso, antes de aplicar migrations em um banco real, é importante revisar o que será executado.

Também é importante lembrar:

  • entidade não deve virar uma classe cheia de regra de tela;
  • configuração de banco deve ficar clara;
  • migrations devem ser revisadas antes de chegar em produção;
  • a connection string não deve ser espalhada pelo código;
  • mudanças no modelo devem ser feitas com intenção.

O que não vamos aprofundar ainda​

Entity Framework possui muitos recursos.

Nesta primeira aula, vimos apenas a entrada.

Ainda não vamos aprofundar:

  • consultas com Include;
  • tracking;
  • relacionamentos mais complexos;
  • performance;
  • transações;
  • seeds;
  • testes com banco.

Esses assuntos aparecem melhor depois que o primeiro fluxo estiver claro.

Resumo prático​

Nesta aula, vimos que:

  • Entity Framework Core é um ORM;
  • ORM mapeia objetos C# para tabelas relacionais;
  • Code First começa pelas classes e configurações do projeto;
  • DbContext representa a comunicação da aplicação com o banco;
  • DbSet representa uma coleção consultável pelo EF Core;
  • IEntityTypeConfiguration separa a configuração de cada entidade;
  • ApplyConfigurationsFromAssembly aplica configurações automaticamente;
  • Microsoft.EntityFrameworkCore.SqlServer conecta o EF Core ao SQL Server;
  • Microsoft.EntityFrameworkCore.Design apoia os comandos de migration;
  • dotnet ef migrations add cria uma migration;
  • dotnet ef database update aplica migrations no banco;
  • o DbContext precisa ser registrado no ASP.NET com AddDbContext.

Fechamento​

O Entity Framework Core permite que a aplicação converse com o banco usando objetos C#.

Na abordagem Code First, partimos das classes da aplicação.

Depois, configuramos como essas classes devem virar tabelas.

Com migrations, o banco passa a acompanhar a evolução do código.

Esse é o primeiro passo para criar uma integração com banco mais orientada a objetos e mais fácil de versionar.
