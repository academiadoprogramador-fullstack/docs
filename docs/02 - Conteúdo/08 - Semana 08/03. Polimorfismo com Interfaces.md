Até aqui, já vimos que classes representam tipos de objetos e que variáveis de referência nos permitem acessar esses objetos na memória. Agora vamos avançar um pouco mais no raciocínio da orientação a objetos.

Quando começamos a modelar um sistema com várias telas, é comum perceber que algumas delas possuem comportamentos em comum. No projeto **Clube da Leitura**, por exemplo, temos várias telas diferentes:

- uma tela para caixas;
- uma tela para revistas;
- uma tela para amigos;
- uma tela para empréstimos.

Embora cada uma tenha suas particularidades, todas participam da navegação do programa de alguma forma. Todas precisam conversar com a `Program`, receber uma opção do usuário e devolver uma informação para que o fluxo continue.

É exatamente nesse tipo de situação que entra o uso de **interfaces**.

---

## O que é uma interface?

Uma interface é um contrato.

Quando dizemos que uma classe implementa uma interface, estamos dizendo que ela se compromete a oferecer determinados comportamentos.

A interface não está preocupada com **como** cada classe vai fazer isso internamente. Ela apenas define **o que** precisa existir.

Veja um exemplo simples:

```cs
public interface ITela
{
    string? ObterEscolhaMenuPrincipal();
}
```

Aqui estamos dizendo o seguinte:

> “Toda classe que quiser ser tratada como uma `ITela` precisa ter um método chamado `ObterEscolhaMenuPrincipal`.”

Perceba que a interface não possui a lógica do método. Ela apenas define a assinatura. A implementação concreta ficará por conta de cada classe.

Essa é a principal ideia da interface: **padronizar a comunicação entre partes diferentes do sistema**.

---

## Por que isso é útil?

Sem interfaces, a classe `Program` precisaria conhecer exatamente o tipo concreto de cada tela com a qual estivesse lidando.

Ou seja, o programa precisaria saber o tempo todo se está trabalhando com:

- `TelaCaixa`
- `TelaRevista`
- `TelaAmigo`
- `TelaEmprestimo`

Isso faz com que a classe principal fique muito acoplada às implementações concretas.

Com interfaces, podemos enxergar todas essas telas por uma mesma perspectiva:

> todas elas são “algum tipo de tela” que consegue devolver uma opção escolhida pelo usuário.

Assim, passamos a trabalhar com a abstração `ITela`, e não diretamente com cada classe concreta o tempo inteiro.

---

## Interfaces e polimorfismo

Esse ponto é muito importante.

O **polimorfismo** acontece quando conseguimos usar referências de um tipo mais genérico para apontar para objetos de tipos mais específicos.

No nosso caso, uma variável do tipo `ITela` pode apontar para objetos diferentes:

- um objeto `TelaCaixa`;
- um objeto `TelaRevista`;
- um objeto `TelaAmigo`;
- um objeto `TelaEmprestimo`.

Ou seja, uma mesma variável referência pode assumir “muitas formas”, desde que todas respeitem o contrato definido pela interface.

É por isso que dizemos que a interface se encaixa diretamente no polimorfismo.

Veja a ideia:

```cs
ITela tela;
```

Essa variável não guarda uma tela específica. Ela guarda uma **referência** para qualquer objeto que implemente `ITela`.

Então poderíamos fazer algo como:

```cs
tela = new TelaCaixa(repositorioCaixa);
```

Ou então:

```cs
tela = new TelaRevista(repositorioRevista, repositorioCaixa);
```

Ou ainda:

```cs
tela = new TelaEmprestimo(repositorioEmprestimo, repositorioRevista, repositorioAmigo);
```

Em todos esses casos, a variável continua sendo do tipo `ITela`. O que muda é o objeto concreto para o qual ela aponta.

Esse é o coração do polimorfismo: **a referência é mais genérica, mas o objeto real continua sendo específico**.

---

## A interface não substitui a classe concreta

É importante não confundir.

A interface não elimina a existência das classes concretas. Quem executa a lógica de verdade continuam sendo as classes reais, como `TelaCaixa`, `TelaRevista` ou `TelaEmprestimo`.

A interface apenas oferece um ponto em comum entre elas.

Pense assim:

- a interface define o contrato;
- a classe concreta realiza o trabalho;
- a variável referência pode apontar para qualquer objeto que respeite aquele contrato.

Isso é muito parecido com o que vimos ao estudar referências de objetos. A variável não carrega “a lógica” em si; ela carrega uma forma de acessar um objeto que está em outro lugar da memória.

## Usando a interface na tela principal

Agora observe o método abaixo:

```cs
public class TelaPrincipal
{
    private RepositorioCaixa repositorioCaixa = new RepositorioCaixa();
    private RepositorioRevista repositorioRevista = new RepositorioRevista();
    private RepositorioAmigo repositorioAmigo = new RepositorioAmigo();
    private RepositorioEmprestimo repositorioEmprestimo = new RepositorioEmprestimo();

    public TelaPrincipal()
    {
        Caixa caixa = new Caixa("Relíquias", "Vermelha", 3);
        repositorioCaixa.Cadastrar(caixa);

        Revista revista = new Revista("Action Comics", 50, 1976, caixa);
        repositorioRevista.Cadastrar(revista);

        Amigo amigo = new Amigo("João", "Seu Jairo", "R. das Flores nº50", "49 98942-9224");
        repositorioAmigo.Cadastrar(amigo);

        Emprestimo emprestimo = new Emprestimo(revista, amigo);
        emprestimo.Abrir();

        repositorioEmprestimo.Cadastrar(emprestimo);
    }

    public ITela? ApresentarMenuOpcoesPrincipal()
    {
        Console.Clear();
        Console.WriteLine("---------------------------------");
        Console.WriteLine("Clube da Leitura");
        Console.WriteLine("---------------------------------");
        Console.WriteLine("1 - Gerenciar caixas de livros");
        Console.WriteLine("2 - Gerenciar revistas");
        Console.WriteLine("3 - Gerenciar amigos");
        Console.WriteLine("4 - Gerenciar empréstimos");
        Console.WriteLine("S - Sair");
        Console.WriteLine("---------------------------------");
        Console.Write("> ");
        string? opcaoMenuPrincipal = Console.ReadLine()?.ToUpper();

        if (opcaoMenuPrincipal == "1")
            return new TelaCaixa(repositorioCaixa);

        if (opcaoMenuPrincipal == "2")
            return new TelaRevista(repositorioRevista, repositorioCaixa);

        if (opcaoMenuPrincipal == "3")
            return new TelaAmigo(repositorioAmigo);

        if (opcaoMenuPrincipal == "4")
            return new TelaEmprestimo(repositorioEmprestimo, repositorioRevista, repositorioAmigo);

        else
            return null;
    }
}
```

Repare com atenção no tipo de retorno do método:

```cs
public ITela? ApresentarMenuOpcoesPrincipal()
```

Esse detalhe é o mais importante de todo o trecho.

O método não retorna `TelaCaixa`, nem `TelaRevista`, nem `TelaAmigo`, nem `TelaEmprestimo` especificamente.

Ele retorna **`ITela`**.

Isso significa que, para quem está chamando esse método, o que importa não é o tipo exato da tela, mas sim o fato de que o objeto retornado se comporta como uma tela válida dentro do programa.

A `TelaPrincipal` escolhe qual objeto concreto instanciar, mas entrega esse objeto para o restante do sistema através da interface.

Essa é uma forma elegante de diminuir acoplamento e aumentar a flexibilidade do código.

## Por que a `Program` precisa disso?

Agora fica mais fácil entender a utilidade prática.

Veja o trecho principal do programa:

```cs
TelaPrincipal telaPrincipal = new();

while (true)
{
    ITela? telaSelecionada = telaPrincipal.ApresentarMenuOpcoesPrincipal();

    if (telaSelecionada == null)
    {
        Console.Clear();
        break;
    }

    while (true)
    {
        string? opcaoMenuInterno;

        opcaoMenuInterno = telaSelecionada.ObterEscolhaMenuPrincipal();

        if (opcaoMenuInterno == "S")
        {
            Console.Clear();
            break;
        }

        if (telaSelecionada is TelaEmprestimo telaEmprestimo)
        {
            if (opcaoMenuInterno == "1")
                telaEmprestimo.Abrir();

            else if (opcaoMenuInterno == "2")
                telaEmprestimo.Concluir();

            else if (opcaoMenuInterno == "3")
                telaEmprestimo.VisualizarTodos(deveExibirCabecalho: true);
        }
        else if (telaSelecionada is TelaBase telaBase)
        {
            if (opcaoMenuInterno == "1")
                telaBase.Cadastrar();

            else if (opcaoMenuInterno == "2")
                telaBase.Editar();

            else if (opcaoMenuInterno == "3")
                telaBase.Excluir();

            else if (opcaoMenuInterno == "4")
                telaBase.VisualizarTodos(deveExibirCabecalho: true);
        }
    }
}
```

A linha abaixo mostra muito bem o polimorfismo em ação:

```cs
ITela? telaSelecionada = telaPrincipal.ApresentarMenuOpcoesPrincipal();
```

Nesse momento, a variável `telaSelecionada` pode receber referências para objetos diferentes. Em uma execução, ela pode apontar para uma `TelaCaixa`. Em outra, pode apontar para uma `TelaRevista`. Em outra, para uma `TelaEmprestimo`.

Mas a `Program` consegue continuar o fluxo sem precisar mudar o tipo da variável a cada caso.

Isso só é possível porque todas essas telas compartilham o contrato definido por `ITela`.

Em seguida, o programa faz isso:

```cs
opcaoMenuInterno = telaSelecionada.ObterEscolhaMenuPrincipal();
```

Perceba a vantagem: a `Program` não precisa saber qual tela concreta está sendo usada para pedir a opção do menu. Ela simplesmente chama o método previsto na interface.

Em outras palavras:

- a `Program` fala com a abstração;
- o objeto concreto responde com seu comportamento específico.

Esse desacoplamento é um dos grandes ganhos da orientação a objetos.

---

## “Passar referências diferentes de telas para a Program”

Agora podemos responder diretamente à ideia principal deste capítulo.

Precisamos usar interfaces porque queremos que a `Program` consiga trabalhar com **referências diferentes de telas** sem depender de uma classe específica toda vez.

Sem a interface, a `Program` teria que receber tipos concretos diferentes em pontos diferentes do código, o que deixaria o fluxo mais engessado e mais difícil de manter.

Com a interface, a `Program` pode receber:

- uma referência para `TelaCaixa`;
- uma referência para `TelaRevista`;
- uma referência para `TelaAmigo`;
- uma referência para `TelaEmprestimo`;

e tratar todas inicialmente como `ITela`.

Ou seja, mudamos o foco de:

> “qual é a classe exata desse objeto?”

para:

> “esse objeto cumpre o contrato que eu preciso?”

Esse raciocínio deixa o sistema mais aberto para crescimento.

Se amanhã surgir uma nova tela, como `TelaMulta`, basta ela implementar `ITela`. A `Program` continuará podendo recebê-la da mesma forma, porque o contrato permanece o mesmo.

---

## Interfaces ajudam a organizar responsabilidades

Outro benefício importante é a separação de responsabilidades.

A `Program` não precisa saber como cada tela monta seu menu, lê o teclado ou organiza suas ações internas. Isso é responsabilidade de cada tela concreta.

Já a interface deixa claro qual é o ponto de comunicação entre a tela e o restante do sistema.

Assim:

- a `Program` controla o fluxo geral;
- a `TelaPrincipal` decide qual tela concreta será aberta;
- cada tela concreta sabe como exibir e tratar seu próprio menu;
- a interface garante que existe um comportamento mínimo comum entre elas.

Perceba como isso melhora a organização do sistema.

---

## Interface é abstração

Se nas classes concretas nós modelamos objetos com detalhes específicos, nas interfaces nós modelamos uma capacidade comum, uma ideia mais abstrata.

`ITela` não representa uma tela específica. Ela representa o conceito de:

> “qualquer objeto que possa funcionar como tela dentro desse fluxo”.

Por isso as interfaces são muito importantes em sistemas orientados a objetos. Elas ajudam a programar contra abstrações, e não contra implementações concretas.

Esse tipo de modelagem costuma deixar o código:

- mais flexível;
- mais reutilizável;
- mais fácil de expandir;
- menos acoplado.

---

## Conclusão

Ao introduzir `ITela`, nosso sistema passa a trabalhar com um contrato comum entre várias telas.

Isso permite que a `Program` receba referências diferentes de objetos concretos e continue funcionando através de uma mesma abstração. É justamente aí que a interface se conecta com o polimorfismo.

Em resumo:

- a interface define o que precisa existir;
- as classes concretas implementam esse comportamento;
- a `Program` usa a interface para trabalhar com diferentes telas sem depender do tipo concreto;
- isso torna o código mais organizado, flexível e orientado a objetos.

Quando usamos interfaces, não estamos apenas “cumprindo uma regra da linguagem”. Estamos modelando melhor a comunicação entre os objetos do sistema.

E, em orientação a objetos, modelar bem essa comunicação faz toda a diferença.
