## TagHelpers no ASP.NET MVC

Até agora, já utilizamos:

- Controllers;
- Views;
- Layouts;
- ViewModels;
- formulários HTML.

Com isso, já conseguimos criar telas, enviar dados para o Controller e navegar entre páginas da aplicação.

Porém, conforme a aplicação cresce, surge outro problema:

> escrever rotas, links e campos de formulário manualmente pode deixar a View repetitiva e mais difícil de manter.

Para resolver esse problema, o ASP.NET MVC possui um recurso chamado **TagHelpers**.

## O problema de escrever HTML manualmente

Imagine que temos um link para a listagem de fabricantes:

```html
<a href="/Fabricante/Listar">Controle de Fabricantes</a>
```

Esse código funciona.

Mas ele depende diretamente do caminho escrito no `href`.

Se o nome do Controller mudar, ou se a rota da aplicação for alterada, esse link pode quebrar.

O mesmo acontece com formulários.

Podemos escrever um formulário assim:

```html
<form action="/Fabricante/Cadastrar" method="post">
    <input type="text" name="Nome">
    <input type="text" name="Email">
    <input type="text" name="Telefone">

    <button type="submit">Cadastrar</button>
</form>
```

Isso também funciona.

Mas precisamos cuidar manualmente de:

- caminho da action;
- método HTTP;
- nomes dos campos;
- vínculo entre input e ViewModel;
- mensagens de validação.

Quanto mais campos uma tela possui, maior fica a chance de erro.

## O que são TagHelpers?

TagHelpers são recursos do ASP.NET MVC que adicionam comportamentos especiais às tags HTML.

Eles permitem escrever HTML normal, mas com atributos que o ASP.NET entende.

Exemplo:

```html
<a asp-controller="Fabricante" asp-action="Listar">Controle de Fabricantes</a>
```

Esse código parece uma tag `<a>` comum.

Mas os atributos `asp-controller` e `asp-action` são interpretados pelo ASP.NET.

Na hora de renderizar a página, o framework gera o `href` correto automaticamente.

## HTML comum e TagHelper

Sem TagHelper:

```html
<a href="/Fabricante/Listar">Controle de Fabricantes</a>
```

Com TagHelper:

```html
<a asp-controller="Fabricante" asp-action="Listar">Controle de Fabricantes</a>
```

O resultado final enviado para o navegador será parecido com:

```html
<a href="/Fabricante/Listar">Controle de Fabricantes</a>
```

Ou seja:

- na View, escrevemos uma referência para Controller e Action;
- no HTML final, o ASP.NET gera o link real.

## Habilitando TagHelpers

Para usar TagHelpers nas Views, normalmente criamos ou editamos o arquivo:

```text
Views/_ViewImports.cshtml
```

Dentro dele, adicionamos:

```html
@addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers
```

Esse arquivo permite importar configurações comuns para várias Views.

Com isso, não precisamos habilitar os TagHelpers manualmente em cada página.

## Substituindo hrefs por ações

Um dos usos mais comuns dos TagHelpers é substituir links escritos manualmente.

Antes:

```html
<a href="/Fabricante/Cadastrar">Cadastrar Fabricante</a>
```

Depois:

```html
<a asp-controller="Fabricante" asp-action="Cadastrar">Cadastrar Fabricante</a>
```

Agora o link aponta para:

- Controller: `FabricanteController`;
- Action: `Cadastrar`.

> Quando usamos `asp-controller="Fabricante"`, não escrevemos a palavra `Controller`.

O ASP.NET já entende que estamos nos referindo ao `FabricanteController`.

## Links para edição

Em algumas telas, precisamos enviar também um parâmetro.

Por exemplo, na edição de um fabricante, normalmente precisamos do `Id`.

Sem TagHelper:

```html
<a href="/Fabricante/Editar/123">Editar</a>
```

Com TagHelper:

```html
<a asp-controller="Fabricante" asp-action="Editar" asp-route-id="123">Editar</a>
```

O atributo `asp-route-id` informa o valor do parâmetro `id`.

Se a Action espera um parâmetro chamado `id`, o ASP.NET monta a rota corretamente.

## Usando valores da ViewModel no link

Em uma listagem, normalmente o `Id` vem da ViewModel.

Exemplo:

```html
@model List<ListarFabricanteViewModel>

@foreach (ListarFabricanteViewModel fabricante in Model)
{
    <a
        asp-controller="Fabricante"
        asp-action="Editar"
        asp-route-id="@fabricante.Id"
    >
        Editar
    </a>
}
```

Nesse caso, cada link de edição recebe o `Id` do fabricante atual.

Assim, o aluno não precisa montar o `href` manualmente com concatenação de texto.

## Links no Layout

TagHelpers também podem ser usados no Layout.

Antes, o menu poderia estar assim:

```html
<nav>
    <a href="/">Início</a>
    <a href="/Fabricante/Listar">Controle de Fabricantes</a>
    <a href="/Equipamento/Listar">Controle de Equipamentos</a>
</nav>
```

Com TagHelpers:

```html
<nav>
    <a asp-controller="Home" asp-action="Index">Início</a>
    <a asp-controller="Fabricante" asp-action="Listar">Controle de Fabricantes</a>
    <a asp-controller="Equipamento" asp-action="Listar">Controle de Equipamentos</a>
</nav>
```

Esse formato deixa o menu mais ligado à estrutura MVC da aplicação.

Em vez de apontar para textos de URL, apontamos para Controllers e Actions.

## TagHelpers em formulários

Outro uso muito importante é nos formulários.

Antes, poderíamos criar um formulário assim:

```html
<form action="/Fabricante/Cadastrar" method="post">
    <label for="Nome">Nome</label>
    <input type="text" id="Nome" name="Nome">

    <label for="Email">Email</label>
    <input type="text" id="Email" name="Email">

    <label for="Telefone">Telefone</label>
    <input type="text" id="Telefone" name="Telefone">

    <button type="submit">Cadastrar</button>
</form>
```

Com TagHelpers, podemos escrever:

```html
@model CadastrarFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Cadastrar" method="post">
    <label asp-for="Nome"></label>
    <input asp-for="Nome">

    <label asp-for="Email"></label>
    <input asp-for="Email">

    <label asp-for="Telefone"></label>
    <input asp-for="Telefone">

    <button type="submit">Cadastrar</button>
</form>
```

Agora o formulário conhece a ViewModel.

Os campos estão ligados às propriedades:

- `Nome`;
- `Email`;
- `Telefone`.

## O atributo asp-for

O atributo `asp-for` liga uma tag HTML a uma propriedade do Model ou da ViewModel.

Exemplo:

```html
<input asp-for="Nome">
```

Se a View usa:

```html
@model CadastrarFabricanteViewModel
```

então `asp-for="Nome"` aponta para:

```csharp
public record CadastrarFabricanteViewModel(
    string Nome,
    string Email,
    string Telefone
);
```

O ASP.NET usa essa informação para gerar atributos como:

- `id`;
- `name`;
- `value`;
- tipo do campo, quando possível.

## Resultado gerado pelo asp-for

Ao escrever:

```html
<input asp-for="Nome">
```

o HTML final pode ficar parecido com:

```html
<input type="text" id="Nome" name="Nome" value="">
```

Isso é importante porque o atributo `name` precisa bater com o nome da propriedade da ViewModel.

É assim que o ASP.NET consegue preencher o objeto recebido no Controller.

## Formulário de cadastro completo

Imagine a Action de cadastro:

```csharp
[HttpGet]
public ActionResult Cadastrar()
{
    return View();
}

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}
```

A View pode ficar assim:

```html
@model CadastrarFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Cadastrar" method="post">
    <div>
        <label asp-for="Nome"></label>
        <input asp-for="Nome">
    </div>

    <div>
        <label asp-for="Email"></label>
        <input asp-for="Email">
    </div>

    <div>
        <label asp-for="Telefone"></label>
        <input asp-for="Telefone">
    </div>

    <button type="submit">Cadastrar</button>
    <a asp-controller="Fabricante" asp-action="Listar">Voltar</a>
</form>
```

Quando o usuário envia o formulário, o ASP.NET cria um objeto `CadastrarFabricanteViewModel`.

Depois, esse objeto é entregue para a Action `Cadastrar`.

## Formulário de edição

Na edição, a ViewModel normalmente possui o `Id`.

Exemplo:

```csharp
public record EditarFabricanteViewModel(
    string Id,
    string Nome,
    string Email,
    string Telefone
);
```

A View de edição pode usar um campo oculto para manter o `Id`:

```html
@model EditarFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Editar" asp-route-id="@Model.Id" method="post">
    <input type="hidden" asp-for="Id">

    <div>
        <label asp-for="Nome"></label>
        <input asp-for="Nome">
    </div>

    <div>
        <label asp-for="Email"></label>
        <input asp-for="Email">
    </div>

    <div>
        <label asp-for="Telefone"></label>
        <input asp-for="Telefone">
    </div>

    <button type="submit">Gravar</button>
    <a asp-controller="Fabricante" asp-action="Listar">Voltar</a>
</form>
```

Aqui usamos duas informações importantes:

- `asp-route-id="@Model.Id"` envia o `Id` na rota;
- `asp-for="Id"` mantém o `Id` também dentro do formulário.

Dependendo da forma como o Controller foi escrito, uma dessas informações pode ser suficiente.

Mas, em telas de edição, é comum manter o identificador disponível.

## Formulário de exclusão

Na exclusão, podemos usar TagHelpers para confirmar a remoção.

Exemplo:

```html
@model ExcluirFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Excluir" asp-route-id="@Model.Id" method="post">
    <p>Deseja realmente excluir o fabricante @Model.Nome?</p>

    <button type="submit">Confirmar</button>
    <a asp-controller="Fabricante" asp-action="Listar">Cancelar</a>
</form>
```

O formulário será enviado para a Action `Excluir`.

O `Id` será enviado junto pela rota.

## TagHelpers e validações

Quando começamos a usar validações nas ViewModels, os TagHelpers também ajudam a exibir mensagens.

Exemplo:

```html
<span asp-validation-for="Nome"></span>
```

Esse TagHelper exibe uma mensagem de validação relacionada à propriedade `Nome`.

Um formulário com validação pode ficar assim:

```html
<div>
    <label asp-for="Nome"></label>
    <input asp-for="Nome">
    <span asp-validation-for="Nome"></span>
</div>
```

Mais adiante, podemos combinar isso com atributos de validação, como `[Required]` e `[StringLength]`.

Por enquanto, o mais importante é entender que o campo, o label e a mensagem podem apontar para a mesma propriedade da ViewModel.

## Principais TagHelpers usados em Views MVC

Os mais comuns neste momento são:

- `asp-controller`, para informar o Controller;
- `asp-action`, para informar a Action;
- `asp-route-id`, para enviar o parâmetro `id`;
- `asp-for`, para ligar uma tag a uma propriedade da ViewModel;
- `asp-validation-for`, para exibir mensagens de validação de um campo;
- `asp-validation-summary`, para exibir um resumo de validações.

Esses atributos ajudam a deixar as Views mais integradas ao MVC.

## Benefícios dos TagHelpers

Utilizar TagHelpers ajuda a criar Views:

- mais fáceis de manter;
- menos dependentes de URLs escritas manualmente;
- mais ligadas aos Controllers e Actions;
- mais integradas com ViewModels;
- mais preparadas para validações.

Eles também deixam o código mais expressivo.

Quando lemos:

```html
<a asp-controller="Fabricante" asp-action="Cadastrar">Cadastrar Fabricante</a>
```

fica claro que o link leva para a Action `Cadastrar` do `FabricanteController`.

## Conclusão

TagHelpers são atributos especiais usados nas Views do ASP.NET MVC.

Eles permitem escrever HTML comum, mas com integração ao framework.

Com eles, conseguimos substituir URLs manuais por referências a:

- Controllers;
- Actions;
- parâmetros de rota.

Também conseguimos criar formulários mais conectados às ViewModels, usando `asp-for`.

Combinando TagHelpers com ViewModels, nossas Views ficam:

- mais claras;
- mais organizadas;
- mais fáceis de alterar;
- mais alinhadas ao padrão MVC.
