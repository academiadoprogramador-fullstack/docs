## ViewModels e Records no ASP.NET MVC

Até agora, já utilizamos:

- Models;
- Views;
- Controllers;
- Layouts;
- ViewBag.

Com isso, já conseguimos criar páginas, receber dados de formulários e exibir informações para o usuário.

Porém, conforme a aplicação cresce, surge um problema importante:

> nem sempre o Model da aplicação é o melhor objeto para ser enviado diretamente para a View.

Para resolver esse problema, utilizamos ViewModels.

E, em C#, uma forma simples de declarar esses objetos é utilizando `record`.

## O problema de usar o Model diretamente

Imagine que temos um `Fabricante`.

Esse Model representa uma entidade do sistema.

Ele pode conter:

- identificador;
- nome;
- e-mail;
- telefone;
- regras de negócio;
- validações;
- outros dados internos.

Em algumas telas, precisamos exibir todos esses dados.

Em outras, precisamos apenas de uma parte deles.

Por exemplo:

- na listagem, queremos mostrar os dados principais;
- no cadastro, não precisamos enviar o `Id`;
- na edição, precisamos do `Id` e dos campos editáveis;
- na exclusão, talvez só queiramos confirmar os dados antes de remover.

Se usarmos sempre o Model diretamente, a View fica muito acoplada à estrutura interna da aplicação.

Isso pode gerar problemas como:

- exposição de dados que a tela não deveria conhecer;
- formulários recebendo campos desnecessários;
- dificuldade para alterar o Model sem quebrar Views;
- mistura entre dados de negócio e dados de apresentação.

## Exemplo sem ViewModel

No Controller, poderíamos retornar o próprio Model para a View:

```cs
[HttpGet]
public ActionResult Listar()
{
    List<Fabricante> fabricantes = repositorioFabricante.SelecionarTodos();

    return View(fabricantes);
}
```

Nesse caso, a View recebe diretamente uma lista de `Fabricante`.

Isso funciona.

Mas significa que a tela passa a depender diretamente da entidade principal do sistema.

Em aplicações pequenas, isso parece simples.

Mas em aplicações maiores, essa dependência dificulta a manutenção.

## O que é uma ViewModel?

Uma ViewModel é uma classe criada especificamente para atender uma View.

Ela representa os dados que aquela tela precisa receber ou enviar.

Em outras palavras:

> a ViewModel é o modelo da tela, não necessariamente o modelo do domínio.

Enquanto o Model representa uma entidade do sistema, a ViewModel representa uma necessidade da interface.

## Model e ViewModel

O Model responde à pergunta:

> "O que existe no sistema?"

A ViewModel responde à pergunta:

> "O que esta tela precisa?"

Por exemplo, o Model `Fabricante` pode representar o fabricante dentro da regra de negócio.

Já uma `CadastrarFabricanteViewModel` representa apenas os campos necessários para cadastrar um fabricante pela tela.

## Criando ViewModels para Fabricante

Podemos criar uma ViewModel para cada operação da tela:

```cs
namespace GestaoDeEquipamentosWeb.ConsoleApp.Models;

public record ListarFabricanteViewModel(
    string Id,
    string Nome,
    string Email,
    string Telefone
);

public record CadastrarFabricanteViewModel(
    string Nome,
    string Email,
    string Telefone
);

public record EditarFabricanteViewModel(
    string Id,
    string Nome,
    string Email,
    string Telefone
);

public record ExcluirFabricanteViewModel(
    string Id,
    string Nome,
    string Email,
    string Telefone
);
```

Cada ViewModel tem apenas os dados necessários para sua respectiva tela.

Observe que a ViewModel de cadastro não possui `Id`.

Isso acontece porque, no cadastro, o usuário ainda não escolhe o identificador.

Esse dado normalmente é gerado pela própria aplicação.

## O que é um record em C#?

O `record` é um tipo do C# usado para representar dados de forma simples.

Ele é parecido com uma classe, mas foi criado especialmente para objetos que carregam valores.

Exemplo:

```cs
public record CadastrarFabricanteViewModel(
    string Nome,
    string Email,
    string Telefone
);
```

Esse código já cria um tipo com as propriedades:

- `Nome`;
- `Email`;
- `Telefone`.

Sem `record`, precisaríamos escrever uma classe mais longa:

```cs
public class CadastrarFabricanteViewModel
{
    public string Nome { get; set; }
    public string Email { get; set; }
    public string Telefone { get; set; }
}
```

Com `record`, conseguimos representar o mesmo conjunto de dados com menos código.

## Por que usar record para ViewModels?

ViewModels normalmente são objetos simples.

Elas carregam dados entre:

- Controller;
- View;
- formulário;
- aplicação.

Na maioria dos casos, elas não precisam conter regras complexas.

Por isso, `record` combina bem com ViewModels.

Ele ajuda a deixar o código:

- mais curto;
- mais direto;
- mais fácil de ler;
- mais focado nos dados da tela.

## Utilizando ViewModel na listagem

No método `Listar`, primeiro buscamos os fabricantes no repositório:

```cs
List<Fabricante> fabricantes = repositorioFabricante.SelecionarTodos();
```

Depois transformamos cada `Fabricante` em uma `ListarFabricanteViewModel`:

```cs
List<ListarFabricanteViewModel> visualizarFabricantes = fabricantes
    .Select(f => new ListarFabricanteViewModel(f.Id, f.Nome, f.Email, f.Telefone))
    .ToList();
```

Por fim, enviamos a lista de ViewModels para a View:

```cs
return View(visualizarFabricantes);
```

O método completo fica assim:

```cs
[HttpGet]
public ActionResult Listar()
{
    List<Fabricante> fabricantes = repositorioFabricante.SelecionarTodos();

    List<ListarFabricanteViewModel> visualizarFabricantes = fabricantes
        .Select(f => new ListarFabricanteViewModel(f.Id, f.Nome, f.Email, f.Telefone))
        .ToList();

    return View(visualizarFabricantes);
}
```

Agora a View não recebe mais a entidade `Fabricante`.

Ela recebe um objeto próprio para listagem.

## Utilizando ViewModel no cadastro

Antes, poderíamos receber cada campo separadamente no Controller:

```cs
[HttpPost]
public ActionResult Cadastrar(string nome, string email, string telefone)
{
    Fabricante novoFabricante = new Fabricante(nome, email, telefone);

    repositorioFabricante.Cadastrar(novoFabricante);

    return RedirectToAction(nameof(Listar));
}
```

Com ViewModel, os dados do formulário são agrupados em um único objeto:

```cs
[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}
```

Esse formato deixa mais claro que aqueles dados pertencem à tela de cadastro.

## Utilizando ViewModel na edição

Na edição, primeiro buscamos o fabricante pelo `Id`:

```cs
Fabricante? fabricante = repositorioFabricante.SelecionarPorId(id);
```

Se ele não existir, voltamos para a listagem:

```cs
if (fabricante == null)
    return RedirectToAction(nameof(Listar));
```

Se existir, criamos uma ViewModel para preencher a tela:

```cs
EditarFabricanteViewModel editarFabricanteVm = new EditarFabricanteViewModel(
    fabricante.Id,
    fabricante.Nome,
    fabricante.Email,
    fabricante.Telefone
);
```

Depois enviamos a ViewModel para a View:

```cs
return View(editarFabricanteVm);
```

## Controller com ViewModels

Com as ViewModels, o Controller passa a fazer uma conversão entre:

- Model;
- ViewModel.

Exemplo:

```cs
public class FabricanteController : Controller
{
    private readonly IRepositorio<Fabricante> repositorioFabricante;

    public FabricanteController()
    {
        ContextoJson contexto = new ContextoJson();
        contexto.Carregar();

        repositorioFabricante = new RepositorioFabricanteEmArquivo(contexto);
    }

    [HttpGet]
    public ActionResult Listar()
    {
        List<Fabricante> fabricantes = repositorioFabricante.SelecionarTodos();

        List<ListarFabricanteViewModel> visualizarFabricantes = fabricantes
            .Select(f => new ListarFabricanteViewModel(f.Id, f.Nome, f.Email, f.Telefone))
            .ToList();

        return View(visualizarFabricantes);
    }

    [HttpGet]
    public ActionResult Cadastrar()
    {
        return View();
    }

    [HttpPost]
    public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
    {
        Fabricante fabricante = new Fabricante(
            cadastrarFabricanteVm.Nome,
            cadastrarFabricanteVm.Email,
            cadastrarFabricanteVm.Telefone
        );

        repositorioFabricante.Cadastrar(fabricante);

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Editar(string id)
    {
        Fabricante? fabricante = repositorioFabricante.SelecionarPorId(id);

        if (fabricante == null)
            return RedirectToAction(nameof(Listar));

        EditarFabricanteViewModel editarFabricanteVm = new EditarFabricanteViewModel(
            fabricante.Id,
            fabricante.Nome,
            fabricante.Email,
            fabricante.Telefone
        );

        return View(editarFabricanteVm);
    }

    [HttpPost]
    public ActionResult Editar(string id, EditarFabricanteViewModel editarFabricanteVm)
    {
        Fabricante fabricanteAtualizado = new Fabricante(
            editarFabricanteVm.Nome,
            editarFabricanteVm.Email,
            editarFabricanteVm.Telefone
        );

        repositorioFabricante.Editar(id, fabricanteAtualizado);

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Excluir(string id)
    {
        Fabricante? fabricante = repositorioFabricante.SelecionarPorId(id);

        if (fabricante == null)
            return RedirectToAction(nameof(Listar));

        ExcluirFabricanteViewModel excluirFabricanteVm = new ExcluirFabricanteViewModel(
            fabricante.Id,
            fabricante.Nome,
            fabricante.Email,
            fabricante.Telefone
        );

        return View(excluirFabricanteVm);
    }

    [HttpPost]
    [ActionName("Excluir")]
    public ActionResult ExcluirConfirmado(string id)
    {
        Fabricante? fabricante = repositorioFabricante.SelecionarPorId(id);

        if (fabricante != null)
            repositorioFabricante.Excluir(fabricante);

        return RedirectToAction(nameof(Listar));
    }
}
```

## Benefícios das ViewModels

Utilizar ViewModels ajuda a criar uma aplicação:

- mais organizada;
- mais segura;
- mais fácil de manter;
- mais alinhada ao padrão MVC.

Também ajuda a separar responsabilidades.

O Model continua cuidando dos dados e regras do sistema.

A ViewModel cuida dos dados que a tela precisa exibir ou receber.

## Conclusão

ViewModels são objetos criados para representar os dados de uma tela.

Elas evitam que a View dependa diretamente dos Models principais da aplicação.

Já os `records` são uma forma simples e curta de criar objetos focados em dados no C#.

Combinando ViewModels e records, conseguimos criar Controllers e Views:

- mais claros;
- mais organizados;
- mais fáceis de alterar;
- mais próximos da ideia de separação de responsabilidades do MVC.
