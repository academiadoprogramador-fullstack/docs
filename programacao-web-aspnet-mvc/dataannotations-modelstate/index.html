DataAnnotations e ModelState

DataAnnotations e ModelState no ASP.NET MVC​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • ViewModels;
  • TagHelpers;
  • formulários.

Com isso, já conseguimos criar telas de cadastro, receber dados enviados pelo usuário e salvar informações na aplicação.

Porém, ainda existe um problema importante:

nem todo dado enviado pelo formulário deve ser aceito pela aplicação.

Por exemplo:

  • o nome não pode estar vazio;
  • o e-mail precisa ter um formato válido;
  • o telefone pode ter um tamanho mínimo;
  • um campo numérico pode ter um valor mínimo ou máximo.

Para lidar com esse tipo de situação, podemos usar DataAnnotations nas ViewModels e o ModelState no Controller.

O problema de confiar no formulário​

Imagine uma tela de cadastro de fabricante.

Ela possui os campos:

  • nome;
  • e-mail;
  • telefone.

Mesmo que a tela mostre esses campos para o usuário, nada garante que os dados enviados serão válidos.

O usuário pode:

  • deixar campos vazios;
  • digitar um e-mail inválido;
  • enviar um texto muito grande;
  • alterar o HTML pelo navegador;
  • enviar uma requisição manualmente.

Por isso, a validação não deve depender apenas da tela.

A aplicação também precisa validar os dados recebidos no servidor.

O que são DataAnnotations?​

DataAnnotations são atributos do C# usados para descrever regras sobre propriedades.

No ASP.NET MVC, elas são muito usadas em ViewModels.

Exemplo:

using System.ComponentModel.DataAnnotations;

public class CadastrarFabricanteViewModel
{
    [Required]
    public string Nome { get; set; }

    [Required]
    [EmailAddress]
    public string Email { get; set; }

    [Required]
    public string Telefone { get; set; }
}

Nesse exemplo, estamos dizendo que:

  • Nome é obrigatório;
  • Email é obrigatório;
  • Email precisa ter formato de e-mail;
  • Telefone é obrigatório.

Essas regras ficam junto da ViewModel.

Assim, a própria ViewModel descreve quais dados a tela espera receber.

Por que validar na ViewModel?​

A ViewModel representa os dados de uma tela.

Por isso, ela é um bom lugar para colocar validações relacionadas à entrada do usuário.

Exemplo:

  • a ViewModel de cadastro pode exigir Nome, Email e Telefone;
  • a ViewModel de edição pode exigir Id, Nome, Email e Telefone;
  • a ViewModel de exclusão pode exigir apenas o Id.

Cada tela possui necessidades diferentes.

Logo, cada ViewModel pode ter suas próprias validações.

ViewModel de cadastro com validações​

Podemos criar uma ViewModel de cadastro assim:

using System.ComponentModel.DataAnnotations;

namespace GestaoDeEquipamentosWeb.ConsoleApp.Models;

public class CadastrarFabricanteViewModel
{
    [Required(ErrorMessage = "O campo Nome é obrigatório.")]
    [StringLength(100, MinimumLength = 2, ErrorMessage = "O campo Nome deve conter entre 2 e 100 caracteres.")]
    public string Nome { get; set; }

    [Required(ErrorMessage = "O campo E-mail é obrigatório.")]
    [EmailAddress(ErrorMessage = "O campo E-mail deve conter um endereço válido.")]
    public string Email { get; set; }

    [Required(ErrorMessage = "O campo Telefone é obrigatório.")]
    [StringLength(20, MinimumLength = 8, ErrorMessage = "O campo Telefone deve conter entre 8 e 20 caracteres.")]
    public string Telefone { get; set; }
}

Agora a ViewModel possui regras mais claras.

Se algum campo não respeitar essas regras, o ASP.NET MVC consegue identificar o problema.

Principais DataAnnotations​

Algumas DataAnnotations comuns são:

  • [Required], para tornar um campo obrigatório;
  • [StringLength], para limitar o tamanho de um texto;
  • [MinLength], para definir um tamanho mínimo;
  • [MaxLength], para definir um tamanho máximo;
  • [EmailAddress], para validar formato de e-mail;
  • [Range], para validar valores numéricos dentro de uma faixa;
  • [Compare], para comparar dois campos.

Cada atributo representa uma regra.

Podemos combinar mais de uma regra na mesma propriedade.

Exemplo com Range​

Imagine uma ViewModel para cadastrar um equipamento.

Ela poderia validar o preço assim:

using System.ComponentModel.DataAnnotations;

public class CadastrarEquipamentoViewModel
{
    [Required(ErrorMessage = "O campo Nome é obrigatório.")]
    public string Nome { get; set; }

    [Range(1, 999999, ErrorMessage = "O preço deve ser maior que zero.")]
    public decimal Preco { get; set; }
}

Nesse caso, Preco precisa estar entre 1 e 999999.

Se o usuário enviar 0, a ViewModel será considerada inválida.

O que é ModelState?​

O ModelState é um objeto do ASP.NET MVC que guarda o estado da validação do Model ou da ViewModel.

Quando uma requisição chega no Controller, o ASP.NET tenta preencher a ViewModel com os dados enviados.

Durante esse processo, ele verifica as DataAnnotations.

Depois disso, o resultado fica disponível em:

ModelState

Podemos perguntar se os dados são válidos usando:

ModelState.IsValid

Se ModelState.IsValid for true, os dados passaram pelas validações.

Se for false, existe pelo menos um erro de validação.

Usando ModelState no cadastro​

Antes, poderíamos cadastrar diretamente:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}

O problema é que esse código cadastra mesmo se os dados estiverem inválidos.

Com ModelState, primeiro verificamos se a ViewModel é válida:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarFabricanteVm);

    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}

Agora o fluxo fica mais seguro.

Se os dados forem inválidos, o Controller devolve a mesma View.

Se os dados forem válidos, o cadastro continua normalmente.

Por que retornar a mesma View?​

Quando fazemos:

return View(cadastrarFabricanteVm);

estamos devolvendo para a tela os dados que o usuário já digitou.

Isso evita que o usuário precise preencher tudo novamente.

Além disso, a View consegue exibir as mensagens de erro geradas pela validação.

Exibindo mensagens na View​

Para exibir erros de validação, usamos TagHelpers.

Exemplo:

@model CadastrarFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Cadastrar" method="post">
    <div>
        <label asp-for="Nome"></label>
        <input asp-for="Nome">
        <span asp-validation-for="Nome"></span>
    </div>

    <div>
        <label asp-for="Email"></label>
        <input asp-for="Email">
        <span asp-validation-for="Email"></span>
    </div>

    <div>
        <label asp-for="Telefone"></label>
        <input asp-for="Telefone">
        <span asp-validation-for="Telefone"></span>
    </div>

    <button type="submit">Cadastrar</button>
</form>

Cada span exibe o erro de uma propriedade específica.

Por exemplo:

  • asp-validation-for="Nome" mostra erros do campo Nome;
  • asp-validation-for="Email" mostra erros do campo Email;
  • asp-validation-for="Telefone" mostra erros do campo Telefone.

Resumo de validações​

Também podemos exibir um resumo com todos os erros da tela.

Para isso, usamos:

<div asp-validation-summary="ModelOnly"></div>

Ou:

<div asp-validation-summary="All"></div>

A opção ModelOnly mostra erros associados ao Model como um todo.

A opção All mostra todos os erros encontrados.

Um exemplo de formulário com resumo:

@model CadastrarFabricanteViewModel

<form asp-controller="Fabricante" asp-action="Cadastrar" method="post">
    <div asp-validation-summary="All"></div>

    <div>
        <label asp-for="Nome"></label>
        <input asp-for="Nome">
        <span asp-validation-for="Nome"></span>
    </div>

    <div>
        <label asp-for="Email"></label>
        <input asp-for="Email">
        <span asp-validation-for="Email"></span>
    </div>

    <div>
        <label asp-for="Telefone"></label>
        <input asp-for="Telefone">
        <span asp-validation-for="Telefone"></span>
    </div>

    <button type="submit">Cadastrar</button>
</form>

Validação na edição​

Na edição, o processo é parecido.

Primeiro, a ViewModel define as regras:

using System.ComponentModel.DataAnnotations;

public class EditarFabricanteViewModel
{
    [Required]
    public string Id { get; set; }

    [Required(ErrorMessage = "O campo Nome é obrigatório.")]
    [StringLength(100, MinimumLength = 2, ErrorMessage = "O campo Nome deve conter entre 2 e 100 caracteres.")]
    public string Nome { get; set; }

    [Required(ErrorMessage = "O campo E-mail é obrigatório.")]
    [EmailAddress(ErrorMessage = "O campo E-mail deve conter um endereço válido.")]
    public string Email { get; set; }

    [Required(ErrorMessage = "O campo Telefone é obrigatório.")]
    [StringLength(20, MinimumLength = 8, ErrorMessage = "O campo Telefone deve conter entre 8 e 20 caracteres.")]
    public string Telefone { get; set; }
}

Depois, o Controller verifica o ModelState:

[HttpPost]
public ActionResult Editar(string id, EditarFabricanteViewModel editarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(editarFabricanteVm);

    Fabricante fabricanteAtualizado = new Fabricante(
        editarFabricanteVm.Nome,
        editarFabricanteVm.Email,
        editarFabricanteVm.Telefone
    );

    repositorioFabricante.Editar(id, fabricanteAtualizado);

    return RedirectToAction(nameof(Listar));
}

Se a ViewModel estiver inválida, a tela de edição é exibida novamente.

Se estiver válida, o registro é atualizado.

Adicionando erros manualmente​

Nem toda validação precisa vir de uma DataAnnotation.

Às vezes, a regra depende de uma consulta na aplicação.

Por exemplo:

não permitir cadastrar dois fabricantes com o mesmo e-mail.

Nesse caso, podemos adicionar um erro manualmente ao ModelState.

Exemplo:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarFabricanteVm);

    bool emailJaExiste = repositorioFabricante
        .SelecionarTodos()
        .Any(f => f.Email == cadastrarFabricanteVm.Email);

    if (emailJaExiste)
    {
        ModelState.AddModelError("Email", "Já existe um fabricante cadastrado com este e-mail.");

        return View(cadastrarFabricanteVm);
    }

    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}

Nesse exemplo, o erro foi associado ao campo Email.

Por isso, ele pode aparecer no trecho:

<span asp-validation-for="Email"></span>

Erros do Model como um todo​

Também podemos adicionar um erro que não pertence a um campo específico.

Para isso, usamos uma string vazia no primeiro parâmetro:

ModelState.AddModelError("", "Não foi possível concluir o cadastro.");

Esse erro pode ser exibido em um resumo de validação:

<div asp-validation-summary="All"></div>

Esse formato é útil quando a mensagem pertence à operação inteira.

Por exemplo:

  • falha ao gravar dados;
  • regra geral de negócio;
  • operação não permitida.

Validação no cliente e no servidor​

Com DataAnnotations, o ASP.NET MVC pode ajudar em dois momentos:

  • validação no cliente;
  • validação no servidor.

A validação no cliente acontece no navegador.

Ela pode impedir o envio do formulário antes da requisição chegar ao servidor.

A validação no servidor acontece no Controller.

Ela é feita com o ModelState.

A validação no servidor é obrigatória. A validação no cliente é apenas uma ajuda para melhorar a experiência do usuário.

Mesmo que exista validação no navegador, o Controller ainda deve verificar ModelState.IsValid.

Fluxo completo da validação​

O fluxo normalmente funciona assim:

  1. O usuário preenche o formulário.
  2. O usuário envia os dados.
  3. O ASP.NET preenche a ViewModel.
  4. O ASP.NET verifica as DataAnnotations.
  5. O Controller consulta ModelState.IsValid.
  6. Se houver erro, o Controller retorna a mesma View.
  7. A View exibe as mensagens de validação.
  8. Se estiver tudo válido, a operação é executada.

Esse fluxo mantém a tela e o Controller trabalhando juntos.

Benefícios das validações com ViewModels​

Utilizar DataAnnotations nas ViewModels ajuda a criar uma aplicação:

  • mais segura;
  • mais previsível;
  • mais fácil de manter;
  • mais clara para o usuário;
  • mais alinhada ao padrão MVC.

O Controller fica responsável por decidir o fluxo.

A ViewModel fica responsável por declarar as regras de entrada da tela.

A View fica responsável por exibir os campos e mensagens.

Conclusão​

DataAnnotations permitem declarar regras de validação diretamente nas ViewModels.

O ModelState permite verificar, dentro do Controller, se os dados enviados respeitam essas regras.

Quando os dados são inválidos, retornamos a mesma View com a ViewModel recebida.

Quando os dados são válidos, seguimos com a operação normalmente.

Combinando ViewModels, DataAnnotations, ModelState e TagHelpers, conseguimos criar formulários:

  • mais seguros;
  • mais claros;
  • mais organizados;
  • mais amigáveis para o usuário;
  • mais próximos da separação de responsabilidades do MVC.
