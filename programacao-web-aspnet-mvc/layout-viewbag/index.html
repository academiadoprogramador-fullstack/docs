Layout e ViewBag

Layout Compartilhado e ViewBag no ASP.NET MVC​

Até agora, já vimos como o ASP.NET MVC organiza a aplicação utilizando:

  • Models;
  • Views;
  • Controllers.

Agora vamos entender outro conceito muito importante em aplicações web:

  • reutilização de interface;
  • compartilhamento de estrutura visual;
  • passagem de dados entre Controller e View.

É exatamente para isso que utilizamos:

  • Layouts
  • ViewBag

O problema da repetição nas Views​

Imagine uma aplicação com várias páginas:

  • página inicial;
  • listagem de equipamentos;
  • cadastro de fabricantes;
  • controle de chamados.

Todas elas normalmente compartilham partes visuais iguais:

  • menu de navegação;
  • cabeçalho;
  • rodapé;
  • imports de CSS;
  • fontes;
  • scripts.

Sem um Layout, precisaríamos repetir isso em todas as Views.

Exemplo:

<html>
  <head>
    ...
  </head>

  <body>
    <nav>...</nav>

    <main>...</main>
  </body>
</html>

Isso gera:

  • duplicação de código;
  • dificuldade de manutenção;
  • inconsistência visual.

O que é um Layout?​

O Layout funciona como um “molde” compartilhado da aplicação.

Ele define a estrutura visual principal do site:

  • cabeçalho;
  • menu;
  • footer;
  • imports;
  • estrutura HTML principal.

As páginas individuais apenas “preenchem” o conteúdo específico.

Em outras palavras:

o Layout define o que é comum entre todas as páginas.

Estrutura do Layout​

No ASP.NET MVC, normalmente o Layout fica em:

Views/Shared/_Layout.cshtml

O nome começa com _ porque ele não representa uma página completa diretamente.

Ele é um arquivo compartilhado entre várias Views.

Exemplo de Layout​

<!DOCTYPE html>
<html lang="pt-BR">
    <head>
        <meta charset="UTF-8">
        <meta name="viewport" content="width=device-width, initial-scale=1.0">

        @* Fonte personalizada *@
        <link rel="preconnect" href="https://fonts.googleapis.com">
        <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
        <link href="https://fonts.googleapis.com/css2?family=Google+Sans:ital,opsz,wght@0,17..18,400..700;1,17..18,400..700&display=swap" rel="stylesheet">

        @* CSS da aplicação *@
        <link rel="stylesheet" href="~/css/styles.css">

        <title>@ViewBag.Titulo - Gestão de Equipamentos</title>
    </head>
    <body>
        <nav>
            <h3>Gestão de Equipamentos</h3>

            <div class="links-navbar">
                <a href="/">Início</a>
                <a href="/Fabricante/Listar">Controle de Fabricantes</a>
                <a href="/Equipamento/Listar">Controle de Equipamentos</a>
                <a href="/Chamado/Listar">Controle de Chamados</a>
            </div>

        </nav>

        <header>
            <h2>@ViewBag.Titulo</h2>
        </header>

        <main>
            @RenderBody()
        </main>
    </body>
</html>

O que é o RenderBody?​

O método:

@RenderBody()

define o local onde o conteúdo da View será inserido.

Pense nele como:

“Aqui dentro será renderizada a página atual.”

Por exemplo:

Se a View atual possuir:

<h1>Cadastro de Equipamentos</h1>

esse conteúdo será inserido dentro do RenderBody().

Relação entre Layout e Views​

O fluxo funciona assim:

  1. O Controller retorna uma View;
  2. A View possui seu conteúdo específico;
  3. O ASP.NET renderiza a View dentro do Layout;
  4. O HTML final é enviado ao navegador.

Ou seja:

  • o Layout fornece a estrutura principal;
  • a View fornece o conteúdo específico.

O que é o ViewBag?​

Agora surge outro problema.

Como passar informações do Controller para a View?

Por exemplo:

  • título da página;
  • mensagens;
  • informações simples de interface.

Uma das formas de fazer isso é utilizando o ViewBag.

ViewBag na prática​

Dentro do Controller:

public IActionResult Listar()
{
    ViewBag.Titulo = "Listagem de Equipamentos";

    return View();
}

Agora a View e o Layout conseguem acessar esse valor:

<h1>@ViewBag.Titulo</h1>

ou:

<title>@ViewBag.Titulo</title>

Como o ViewBag funciona?​

O ViewBag é um objeto dinâmico criado pelo ASP.NET MVC.

Ele permite enviar dados do Controller para a View sem precisar criar uma classe específica.

Exemplo:

ViewBag.Mensagem = "Cadastro realizado com sucesso";
ViewBag.Usuario = "João";

Na View:

<p>@ViewBag.Mensagem</p>

Quando usar ViewBag?​

O ViewBag é útil para:

  • dados simples;
  • títulos;
  • mensagens;
  • informações auxiliares da interface.

Porém, ele não é ideal para dados complexos.

Para estruturas maiores normalmente utilizamos:

  • Models;
  • ViewModels.

Layout e Reaproveitamento​

O principal benefício do Layout é a reutilização.

Sem ele:

  • cada página precisaria repetir HTML;
  • qualquer alteração teria que ser feita manualmente em várias Views.

Com Layout:

  • alteramos uma única vez;
  • todas as páginas são atualizadas automaticamente.

Separação de Responsabilidades​

Assim como o MVC separa:

  • Model;
  • View;
  • Controller;

o Layout também ajuda a separar responsabilidades.

Ele centraliza:

  • aparência;
  • estrutura visual;
  • navegação.

Enquanto as Views focam apenas no conteúdo específico.

Conclusão​

Layouts permitem criar uma estrutura visual compartilhada entre várias páginas da aplicação.

Eles ajudam a:

  • evitar duplicação;
  • manter consistência visual;
  • facilitar manutenção.

Já o ViewBag permite enviar dados simples do Controller para as Views e Layouts.

Combinando os dois conceitos, conseguimos criar aplicações web:

  • mais organizadas;
  • mais reutilizáveis;
  • mais fáceis de manter;
  • mais alinhadas ao padrão MVC.
