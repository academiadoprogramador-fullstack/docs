TempData no ASP.NET MVC

TempData no ASP.NET MVC​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • Layouts;
  • ViewBag;
  • ViewModels;
  • validações com ModelState;
  • redirecionamentos com RedirectToAction.

Com isso, já conseguimos criar telas, receber dados de formulários, validar informações e redirecionar o usuário depois de uma operação.

Porém, ainda existe um problema comum em aplicações web:

como exibir uma mensagem depois de redirecionar o usuário para outra Action?

Por exemplo:

  • "Fabricante cadastrado com sucesso.";
  • "Registro editado com sucesso.";
  • "Não foi possível encontrar o item selecionado.";
  • "Equipamento excluído com sucesso.".

Para esse tipo de situação, o ASP.NET MVC possui um recurso chamado TempData.

O problema das mensagens após um redirecionamento​

Imagine uma Action de cadastro.

Depois de salvar o fabricante, queremos voltar para a listagem:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarFabricanteVm);

    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    return RedirectToAction(nameof(Listar));
}

Esse fluxo é correto.

Após cadastrar, o usuário é redirecionado para a tela de listagem.

Mas agora surge uma dúvida:

onde guardar a mensagem de sucesso para a próxima tela conseguir exibir?

Por que ViewBag não resolve esse caso?​

Poderíamos tentar usar ViewBag:

ViewBag.Mensagem = "Fabricante cadastrado com sucesso.";

return RedirectToAction(nameof(Listar));

Esse código parece fazer sentido.

Mas ele não funciona como esperado.

O motivo é simples:

o ViewBag vive apenas durante a requisição atual.

Quando usamos RedirectToAction, o navegador faz uma nova requisição.

Ou seja:

  1. O usuário envia o formulário.
  2. A Action Cadastrar processa o cadastro.
  3. O Controller retorna um redirecionamento.
  4. O navegador faz uma nova requisição para Listar.
  5. A Action Listar executa em outra requisição.

O ViewBag criado na Action Cadastrar não chega até a Action Listar.

Ele desaparece no fim da requisição anterior.

O que é TempData?​

O TempData é um recurso do ASP.NET MVC usado para guardar dados temporários entre requisições.

Ele é muito útil quando uma Action precisa enviar uma informação para outra Action após um redirecionamento.

Em outras palavras:

TempData serve para mensagens ou dados pequenos que precisam sobreviver a um RedirectToAction.

O uso mais comum é exibir mensagens de sucesso, erro ou aviso depois de uma operação.

Usando TempData no Controller​

No cadastro, podemos guardar uma mensagem no TempData antes de redirecionar:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarFabricanteVm);

    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    TempData["MensagemSucesso"] = "Fabricante cadastrado com sucesso.";

    return RedirectToAction(nameof(Listar));
}

Agora a mensagem fica disponível na próxima requisição.

Nesse caso, ela poderá ser lida na tela de listagem.

Lendo TempData na View​

Na View de listagem, podemos verificar se existe uma mensagem:

@if (TempData["MensagemSucesso"] is not null) {
<div class="mensagem-sucesso">@TempData["MensagemSucesso"]</div>
}

Se a mensagem existir, ela será exibida.

Se não existir, nada será mostrado.

Esse formato é comum para mensagens temporárias.

Elas aparecem uma vez e depois deixam de existir.

Exemplo com mensagem de erro​

O TempData também pode ser usado para mensagens de erro.

Imagine que o usuário tentou editar um fabricante que não existe:

[HttpGet]
public ActionResult Editar(string id)
{
    Fabricante? fabricante = repositorioFabricante.SelecionarPorId(id);

    if (fabricante == null)
    {
        TempData["MensagemErro"] = "Fabricante não encontrado.";

        return RedirectToAction(nameof(Listar));
    }

    EditarFabricanteViewModel editarFabricanteVm = new EditarFabricanteViewModel(
        fabricante.Id,
        fabricante.Nome,
        fabricante.Email,
        fabricante.Telefone
    );

    return View(editarFabricanteVm);
}

Nesse exemplo, a Action não consegue continuar para a tela de edição.

Por isso, ela volta para a listagem e envia uma mensagem temporária.

Exibindo mensagens diferentes​

Na View, podemos exibir mensagens de sucesso e erro separadamente:

@if (TempData["MensagemSucesso"] is not null) {
<div class="mensagem-sucesso">@TempData["MensagemSucesso"]</div>
} @if (TempData["MensagemErro"] is not null) {
<div class="mensagem-erro">@TempData["MensagemErro"]</div>
}

Assim, a tela consegue mostrar mensagens com estilos diferentes.

Por exemplo:

  • sucesso em verde;
  • erro em vermelho;
  • aviso em amarelo.

Usando TempData no Layout​

Em muitas aplicações, as mensagens aparecem sempre no mesmo lugar da página.

Por exemplo, logo abaixo do menu ou antes do conteúdo principal.

Nesses casos, podemos colocar a exibição no Layout.

Exemplo:

<body>
  <nav>
    <h3>Gestão de Equipamentos</h3>
  </nav>

  <main>
    @if (TempData["MensagemSucesso"] is not null) {
    <div class="mensagem-sucesso">@TempData["MensagemSucesso"]</div>
    } @if (TempData["MensagemErro"] is not null) {
    <div class="mensagem-erro">@TempData["MensagemErro"]</div>
    } @RenderBody()
  </main>
</body>

Com isso, qualquer Controller pode criar uma mensagem usando TempData.

O Layout fica responsável por exibir essa mensagem.

Isso evita repetir o mesmo código em várias Views.

TempData aparece apenas uma vez​

Uma característica importante do TempData é que ele foi feito para dados temporários.

Depois que um valor é lido, ele normalmente é marcado para remoção.

Na prática, isso significa que a mensagem aparece uma vez.

Esse comportamento combina muito bem com mensagens após uma operação:

  • o usuário cadastra um registro;
  • a listagem mostra "Cadastro realizado com sucesso";
  • ao atualizar a página novamente, a mensagem desaparece.

Essa mensagem é chamada muitas vezes de flash message.

Ou seja, uma mensagem rápida, exibida apenas no próximo carregamento da página.

Mantendo um valor por mais tempo​

Em alguns casos, podemos querer ler um valor do TempData sem removê-lo imediatamente.

Para isso, o ASP.NET MVC possui o método Peek.

Exemplo:

string? mensagem = TempData.Peek("MensagemSucesso") as string;

Também existe o método Keep, usado para manter valores depois da leitura.

Exemplo:

TempData.Keep("MensagemSucesso");

Esses recursos existem, mas devem ser usados com cuidado.

Na maioria dos casos, para mensagens de sucesso ou erro, o comportamento padrão do TempData já é suficiente.

Quando usar TempData?​

Use TempData quando a informação precisa passar de uma requisição para outra.

Principalmente quando existe um redirecionamento.

Exemplos comuns:

  • mensagem após cadastrar;
  • mensagem após editar;
  • mensagem após excluir;
  • aviso ao redirecionar para outra tela;
  • erro simples ao voltar para uma listagem;
  • confirmação temporária depois de uma ação do usuário.

Um bom sinal de que TempData faz sentido é a presença de:

return RedirectToAction(...);

Se a mensagem deve aparecer depois desse redirecionamento, TempData costuma ser a escolha correta.

Quando usar ViewBag?​

Use ViewBag quando a informação será usada apenas na View retornada pela requisição atual.

Exemplos comuns:

  • título da página;
  • subtítulo;
  • texto auxiliar;
  • lista simples para montar uma parte da interface;
  • informação pequena usada pelo Layout na mesma requisição.

Exemplo:

[HttpGet]
public ActionResult Listar()
{
    ViewBag.Titulo = "Listagem de Fabricantes";

    List<Fabricante> fabricantes = repositorioFabricante.SelecionarTodos();

    return View(fabricantes);
}

Nesse caso, o Controller retorna a View diretamente.

Não existe redirecionamento.

Então o ViewBag.Titulo estará disponível para a View e para o Layout.

TempData vs ViewBag​

A diferença principal está no tempo de vida da informação.

ViewBag existe apenas durante a requisição atual.

TempData pode sobreviver até a próxima requisição.

Comparação prática:

SituaçãoMelhor opção
Enviar título para a View atualViewBag
Mostrar mensagem depois de return View()ViewBag ou ViewModel
Mostrar mensagem depois de RedirectToAction()TempData
Enviar dados principais da telaViewModel
Enviar lista de registros para uma tabelaViewModel
Exibir mensagem de sucesso após cadastrarTempData
Exibir mensagem de erro após redirecionarTempData

Uma forma simples de decidir é perguntar:

a informação precisa sobreviver a um redirecionamento?

Se a resposta for sim, use TempData.

Se a resposta for não, ViewBag pode ser suficiente para dados simples.

TempData não substitui ViewModel​

Apesar de ser útil, TempData não deve ser usado para carregar os dados principais de uma tela.

Evite usar TempData para enviar:

  • listas grandes;
  • objetos complexos;
  • dados de formulário inteiro;
  • informações importantes para a regra de negócio.

Para esses casos, prefira ViewModels.

O TempData deve ficar reservado para informações pequenas e passageiras.

Exemplo adequado:

TempData["MensagemSucesso"] = "Equipamento excluído com sucesso.";

Exemplo inadequado:

TempData["TodosOsEquipamentos"] = equipamentos;

No segundo caso, a listagem de equipamentos deveria ser buscada no Controller e enviada para a View por uma ViewModel.

Fluxo completo com TempData​

O fluxo mais comum fica assim:

  1. O usuário envia um formulário.
  2. O Controller valida os dados com ModelState.
  3. A operação é executada.
  4. O Controller guarda uma mensagem no TempData.
  5. O Controller redireciona com RedirectToAction.
  6. A próxima Action é executada.
  7. A View ou o Layout lê o TempData.
  8. A mensagem é exibida para o usuário.

Esse fluxo mantém a aplicação organizada.

O Controller decide o resultado da operação.

O TempData carrega a mensagem temporária.

A View ou o Layout apenas exibem essa mensagem.

Exemplo completo no cadastro​

O Controller pode ficar assim:

[HttpPost]
public ActionResult Cadastrar(CadastrarFabricanteViewModel cadastrarFabricanteVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarFabricanteVm);

    Fabricante fabricante = new Fabricante(
        cadastrarFabricanteVm.Nome,
        cadastrarFabricanteVm.Email,
        cadastrarFabricanteVm.Telefone
    );

    repositorioFabricante.Cadastrar(fabricante);

    TempData["MensagemSucesso"] = "Fabricante cadastrado com sucesso.";

    return RedirectToAction(nameof(Listar));
}

E o Layout pode exibir a mensagem assim:

@if (TempData["MensagemSucesso"] is not null) {
<div class="mensagem-sucesso">@TempData["MensagemSucesso"]</div>
}

Com isso, a mensagem será exibida na próxima página carregada.

Depois disso, ela deixa de aparecer.

Conclusão​

O TempData permite enviar dados temporários entre requisições no ASP.NET MVC.

Ele é especialmente útil quando usamos RedirectToAction.

Já o ViewBag serve para dados simples usados apenas na requisição atual.

Na prática:

  • use ViewBag para títulos e informações auxiliares da View atual;
  • use TempData para mensagens temporárias após redirecionamentos;
  • use ViewModels para os dados principais da tela.

Com essa separação, o Controller fica mais claro e as Views recebem cada tipo de informação pelo recurso mais adequado.
