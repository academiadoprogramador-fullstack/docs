AutoMapper

AutoMapper no ASP.NET MVC​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • ViewModels;
  • DTOs;
  • Services;
  • Extension Methods;
  • ModelState;
  • TempData.

Com isso, já conseguimos organizar melhor o fluxo da aplicação.

O Controller recebe dados da tela por ViewModels.

O Service recebe dados por DTOs.

As Views também recebem ViewModels para exibir informações ao usuário.

Porém, conforme a aplicação cresce, aparece outro problema comum:

muitas vezes precisamos copiar dados de um objeto para outro.

Por exemplo:

  • transformar uma CadastrarCaixaViewModel em CadastrarCaixaDto;
  • transformar um DetalhesCaixaDto em EditarCaixaViewModel;
  • transformar uma lista de ListarCaixasDto em uma lista de ListarCaixasViewModel.

Essa conversão entre objetos é chamada de mapeamento.

Para facilitar esse trabalho, podemos usar uma biblioteca chamada AutoMapper.

O problema do mapeamento manual​

Imagine uma Action de cadastro.

A View envia uma CadastrarCaixaViewModel.

Mas o ServicoCaixa espera receber uma CadastrarCaixaDto.

Sem AutoMapper, poderíamos fazer a conversão manualmente:

CadastrarCaixaDto dto = new CadastrarCaixaDto(
    cadastrarVm.Etiqueta,
    cadastrarVm.Cor,
    cadastrarVm.DiasDeEmprestimo
);

Esse código funciona.

Mas ele começa a se repetir em várias Actions.

Na edição, também precisamos converter:

EditarCaixaDto dto = new EditarCaixaDto(
    editarVm.Id,
    editarVm.Etiqueta,
    editarVm.Cor,
    editarVm.DiasDeEmprestimo
);

Na listagem, a conversão pode envolver uma lista:

List<ListarCaixasViewModel> listarVms = dtos
    .Select(dto => new ListarCaixasViewModel(
        dto.Id,
        dto.Etiqueta,
        dto.Cor,
        dto.DiasDeEmprestimo
    ))
    .ToList();

Perceba que essas conversões não representam uma regra importante da aplicação.

Elas apenas copiam valores entre objetos com estruturas parecidas.

Quando esse tipo de código se espalha pelo Controller, as Actions ficam maiores e menos focadas no fluxo da requisição.

O que é AutoMapper?​

O AutoMapper é uma biblioteca usada para mapear dados de um objeto para outro.

Em outras palavras:

AutoMapper ajuda a transformar um objeto de origem em um objeto de destino.

Por exemplo:

CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

Nesse exemplo:

  • cadastrarVm é o objeto de origem;
  • CadastrarCaixaDto é o tipo de destino;
  • mapeador.Map executa a conversão.

O AutoMapper compara as propriedades dos objetos.

Quando os nomes e tipos são compatíveis, ele consegue copiar os valores automaticamente.

Mapeamento entre ViewModel e DTO​

No Clube da Leitura Web, usamos objetos diferentes para responsabilidades diferentes.

Por exemplo:

TipoResponsabilidade
ViewModelRepresentar os dados da tela
DTOTransportar dados entre Controller e Service
EntidadeRepresentar conceitos do domínio

Isso deixa o projeto mais organizado.

Mas também cria a necessidade de converter dados entre esses objetos.

No cadastro de caixa, o fluxo fica assim:

  1. A View envia uma CadastrarCaixaViewModel.
  2. O Controller valida o ModelState.
  3. O Controller transforma a ViewModel em CadastrarCaixaDto.
  4. O Controller envia o DTO para o ServicoCaixa.
  5. O Service executa a operação.

Com AutoMapper, a etapa de conversão fica menor:

CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

O que é um Profile?​

Para o AutoMapper saber quais conversões podem ser feitas, precisamos criar um Profile.

Um Profile é uma classe onde registramos os mapas da aplicação.

Cada mapa diz:

objetos deste tipo podem ser convertidos para objetos daquele tipo.

No módulo de caixas, podemos criar um Profile assim:

public class MapeamentoCaixaProfile : Profile
{
    public MapeamentoCaixaProfile()
    {
        CreateMap<ListarCaixasDto, ListarCaixasViewModel>();
        CreateMap<CadastrarCaixaViewModel, CadastrarCaixaDto>();
        CreateMap<EditarCaixaViewModel, EditarCaixaDto>();
        CreateMap<DetalhesCaixaDto, EditarCaixaViewModel>();
        CreateMap<DetalhesCaixaDto, ExcluirCaixaViewModel>();
    }
}

Observe que a classe herda de Profile.

Essa classe vem do AutoMapper.

Dentro do construtor, usamos CreateMap para registrar as conversões.

Entendendo o CreateMap​

O método CreateMap recebe dois tipos.

O primeiro é o tipo de origem.

O segundo é o tipo de destino.

Exemplo:

CreateMap<CadastrarCaixaViewModel, CadastrarCaixaDto>();

Esse mapa significa:

uma CadastrarCaixaViewModel pode ser convertida para uma CadastrarCaixaDto.

Depois disso, o Controller pode fazer:

CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

Outro exemplo:

CreateMap<DetalhesCaixaDto, EditarCaixaViewModel>();

Esse mapa significa:

um DetalhesCaixaDto pode ser convertido para uma EditarCaixaViewModel.

Isso é útil quando o usuário acessa a tela de edição.

O Service busca os dados da caixa.

O Controller transforma esses dados em uma ViewModel própria para a tela.

Quando o AutoMapper consegue mapear automaticamente​

O AutoMapper funciona melhor quando os objetos possuem propriedades com nomes compatíveis.

Por exemplo:

public record CadastrarCaixaViewModel(
    string Etiqueta,
    string Cor,
    int DiasDeEmprestimo
);

public record CadastrarCaixaDto(
    string Etiqueta,
    string Cor,
    int DiasDeEmprestimo
);

Nesse caso, os nomes são iguais:

  • Etiqueta;
  • Cor;
  • DiasDeEmprestimo.

Por isso, o AutoMapper consegue copiar os valores sem configuração extra.

Se os nomes forem muito diferentes, será necessário configurar o mapeamento com mais detalhes.

Para o nosso momento, vamos manter os nomes alinhados entre ViewModels e DTOs sempre que fizer sentido.

Registrando o AutoMapper no Program.cs​

Criar o Profile não é suficiente.

Também precisamos registrar o AutoMapper no container de injeção de dependência.

Esse registro fica no Program.cs:

builder.Services.AddAutoMapper(configuracao =>
{
    configuracao.AddProfile<MapeamentoCaixaProfile>();
});

Esse código diz ao ASP.NET:

registre o AutoMapper e carregue as configurações do MapeamentoCaixaProfile.

Depois disso, o IMapper pode ser injetado no Controller.

Injetando IMapper no Controller​

Assim como fazemos com Services e repositórios, também podemos receber o AutoMapper por injeção de dependência.

No Controller, usamos a interface IMapper:

public class CaixaController(ServicoCaixa servicoCaixa, IMapper mapeador) : Controller
{
}

Nesse exemplo, o Controller recebe duas dependências:

  • ServicoCaixa, para executar as operações da aplicação;
  • IMapper, para converter objetos.

O Controller não cria o AutoMapper manualmente.

Ele apenas declara que precisa de um mapeador.

O ASP.NET entrega esse objeto porque ele foi registrado no Program.cs.

Usando AutoMapper na listagem​

Na listagem, o Service retorna DTOs:

List<ListarCaixasDto> dtos = servicoCaixa.SelecionarTodos();

Mas a View deve receber ViewModels:

List<ListarCaixasViewModel> listarVms =
    mapeador.Map<List<ListarCaixasViewModel>>(dtos);

Esse código transforma uma lista de DTOs em uma lista de ViewModels.

Para isso funcionar, o Profile precisa ter o mapa entre os itens da lista:

CreateMap<ListarCaixasDto, ListarCaixasViewModel>();

Não precisamos criar um mapa específico de List<ListarCaixasDto> para List<ListarCaixasViewModel>.

O AutoMapper entende que, se sabe converter um item, também consegue converter uma lista desses itens.

Usando AutoMapper no cadastro​

Na Action de cadastro, recebemos uma ViewModel:

public ActionResult Cadastrar(CadastrarCaixaViewModel cadastrarVm)

Depois da validação do ModelState, convertemos a ViewModel para DTO:

CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

Em seguida, enviamos o DTO para o Service:

Result resultado = servicoCaixa.Cadastrar(dto);

O Controller continua responsável pelo fluxo da tela.

O AutoMapper apenas reduz o código de cópia entre objetos.

Usando AutoMapper na edição​

Na tela de edição, temos dois momentos de mapeamento.

Primeiro, no HttpGet, o Controller busca os detalhes da caixa:

Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

Se a busca funcionar, o DTO é convertido para a ViewModel da tela:

EditarCaixaViewModel editarVm =
    mapeador.Map<EditarCaixaViewModel>(resultado.Value);

Depois, no HttpPost, a ViewModel preenchida pelo usuário é convertida para DTO:

EditarCaixaDto dto = mapeador.Map<EditarCaixaDto>(editarVm);

Cada conversão tem uma direção.

Por isso, o Profile possui dois mapas relacionados à edição:

CreateMap<DetalhesCaixaDto, EditarCaixaViewModel>();
CreateMap<EditarCaixaViewModel, EditarCaixaDto>();

Usando AutoMapper na exclusão​

Na exclusão, o fluxo é parecido com a edição.

O Controller busca os detalhes da caixa:

Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

Depois transforma o DTO em uma ViewModel de exclusão:

ExcluirCaixaViewModel excluirVm =
    mapeador.Map<ExcluirCaixaViewModel>(resultado.Value);

Para isso, o Profile possui este mapa:

CreateMap<DetalhesCaixaDto, ExcluirCaixaViewModel>();

Assim, a tela de exclusão recebe apenas os dados que ela precisa mostrar.

Controller completo usando AutoMapper​

Com o AutoMapper configurado, o Controller fica mais direto.

Observe principalmente as chamadas para:

mapeador.Map<...>(...)
namespace ClubeDaLeituraWeb.WebApp.ModuloCaixa.Apresentacao;

public class CaixaController(ServicoCaixa servicoCaixa, IMapper mapeador) : Controller
{
    [HttpGet]
    public ActionResult Listar()
    {
        List<ListarCaixasDto> dtos = servicoCaixa.SelecionarTodos();

        List<ListarCaixasViewModel> listarVms = mapeador.Map<List<ListarCaixasViewModel>>(dtos);

        return View(listarVms);
    }

    [HttpGet]
    public ActionResult Cadastrar()
    {
        CadastrarCaixaViewModel cadastrarVm = new CadastrarCaixaViewModel(
            string.Empty,
            string.Empty,
            7
        );

        return View(cadastrarVm);
    }

    [HttpPost]
    public ActionResult Cadastrar(CadastrarCaixaViewModel cadastrarVm)
    {
        if (!ModelState.IsValid)
            return View(cadastrarVm);

        CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

        Result resultado = servicoCaixa.Cadastrar(dto);

        if (resultado.IsFailed)
        {
            ModelState.AdicionarErros(resultado);

            return View(cadastrarVm);
        }

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Editar(string id)
    {
        Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

        if (resultado.IsFailed)
        {
            TempData.AdicionarErro(resultado.ObterMensagemErro());

            return RedirectToAction(nameof(Listar));
        }

        EditarCaixaViewModel editarVm =
            mapeador.Map<EditarCaixaViewModel>(resultado.Value);

        return View(editarVm);
    }

    [HttpPost]
    public ActionResult Editar(EditarCaixaViewModel editarVm)
    {
        if (!ModelState.IsValid)
            return View(editarVm);

        EditarCaixaDto dto = mapeador.Map<EditarCaixaDto>(editarVm);

        Result resultado = servicoCaixa.Editar(dto);

        if (resultado.IsFailed)
        {
            ModelState.AdicionarErros(resultado);

            return View(editarVm);
        }

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Excluir(string id)
    {
        Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

        if (resultado.IsFailed)
        {
            TempData.AdicionarErro(resultado.ObterMensagemErro());

            return RedirectToAction(nameof(Listar));
        }

        ExcluirCaixaViewModel excluirVm =
            mapeador.Map<ExcluirCaixaViewModel>(resultado.Value);

        return View(excluirVm);
    }

    [HttpPost]
    public ActionResult Excluir(ExcluirCaixaViewModel excluirVm)
    {
        Result resultado = servicoCaixa.Excluir(excluirVm.Id);

        if (resultado.IsFailed)
            TempData.AdicionarErro(resultado.ObterMensagemErro());

        return RedirectToAction(nameof(Listar));
    }
}

O Controller ainda precisa conhecer os tipos envolvidos no fluxo.

Mas ele não precisa escrever manualmente cada propriedade copiada.

Onde colocar os Profiles​

Uma organização possível é deixar os Profiles perto do módulo que eles configuram.

Por exemplo:

ModuloCaixa
+ Apresentacao
  + CaixaController.cs
  + MapeamentoCaixaProfile.cs
  + ViewModels
+ Aplicacao
  + ServicoCaixa.cs
  + Dtos

Essa estrutura deixa claro que o MapeamentoCaixaProfile está relacionado ao módulo de caixas.

Em projetos maiores, também é comum existir uma pasta específica para mapeamentos.

O mais importante é manter um padrão claro no projeto.

AutoMapper não substitui a separação de camadas​

O AutoMapper não deve ser usado para colocar regras escondidas dentro do mapeamento.

Ele deve ser usado principalmente para conversões simples.

Por exemplo:

  • copiar Etiqueta para Etiqueta;
  • copiar Cor para Cor;
  • copiar DiasDeEmprestimo para DiasDeEmprestimo;
  • converter uma lista de DTOs para uma lista de ViewModels.

Regras como verificar etiqueta duplicada continuam no Service.

Validações da tela continuam no ModelState.

Mensagens temporárias continuam no TempData.

O AutoMapper apenas ajuda a reduzir o código repetitivo de transformação entre objetos.

Erros comuns ao usar AutoMapper​

Um erro comum é esquecer de registrar o Profile no Program.cs.

Nesse caso, o Controller chama:

mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

Mas o AutoMapper não sabe como converter esses tipos.

Outro erro comum é criar o mapa na direção errada.

Por exemplo:

CreateMap<CadastrarCaixaDto, CadastrarCaixaViewModel>();

Esse mapa permite converter de DTO para ViewModel.

Mas, se o Controller precisa converter de ViewModel para DTO, o mapa correto é:

CreateMap<CadastrarCaixaViewModel, CadastrarCaixaDto>();

A direção importa.

Por isso, sempre pense:

qual é o objeto que eu tenho agora e qual objeto eu preciso obter?

Quando usar AutoMapper​

Use AutoMapper quando:

  • os objetos possuem propriedades parecidas;
  • a conversão é simples;
  • o mapeamento aparece em várias Actions;
  • o Controller está ficando cheio de código de cópia;
  • você quer padronizar conversões entre ViewModels e DTOs.

Ele combina muito bem com a separação entre camada de apresentação e camada de aplicação.

No nosso caso, ele ajuda o Controller a conversar com o Service sem misturar ViewModels com DTOs.

Quando evitar AutoMapper​

Evite usar AutoMapper quando a conversão tem muita regra de negócio.

Também evite usá-lo para esconder operações difíceis de entender.

Se a transformação exige decisões importantes, talvez seja melhor escrever o código manualmente.

Assim, a regra fica explícita para quem está lendo.

AutoMapper é uma boa ferramenta para reduzir repetição.

Mas clareza continua sendo mais importante do que escrever menos código.

Conclusão​

AutoMapper é uma biblioteca que ajuda a converter objetos de um tipo para outro.

No ASP.NET MVC, ele é muito útil para mapear:

  • ViewModels para DTOs;
  • DTOs para ViewModels;
  • listas de DTOs para listas de ViewModels.

Para usá-lo, criamos um Profile com os mapas necessários.

Depois registramos esse Profile no Program.cs.

Por fim, recebemos IMapper no Controller e usamos o método Map.

Com isso, o Controller fica mais limpo.

O Service continua cuidando das regras da aplicação.

As ViewModels continuam representando as telas.

E os DTOs continuam transportando dados entre a apresentação e a aplicação.
