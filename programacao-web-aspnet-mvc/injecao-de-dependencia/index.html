Injeção de Dependência no ASP.NET

Injeção de Dependência no ASP.NET​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • ViewModels;
  • validações;
  • repositórios;
  • arquivos para persistência de dados.

Com isso, já conseguimos criar telas e executar operações dentro da aplicação.

Porém, conforme o projeto cresce, aparece um problema importante:

como um Controller deve acessar as classes que ele precisa para trabalhar?

Por exemplo, um Controller de caixas de revista precisa acessar um repositório.

Esse repositório precisa acessar o contexto de arquivos.

Se cada Controller começar a criar essas dependências manualmente, o código fica mais acoplado e mais difícil de manter.

Para resolver esse problema, o ASP.NET utiliza Injeção de Dependência.

O problema de criar dependências manualmente​

Imagine um Controller que precisa de um repositório.

Sem injeção de dependência, poderíamos escrever algo assim:

public class CaixaController : Controller
{
    private readonly IRepositorio<Caixa> repositorioCaixa;

    public CaixaController()
    {
        ContextoJson contexto = new ContextoJson();
        contexto.Carregar();

        repositorioCaixa = new RepositorioCaixaEmArquivo(contexto);
    }
}

Esse código funciona.

Mas o Controller passou a conhecer detalhes demais:

  • ele sabe que os dados vêm de um arquivo JSON;
  • ele sabe qual classe concreta de repositório será usada;
  • ele sabe como criar o contexto;
  • ele precisa repetir essa criação em outros Controllers.

Isso deixa o Controller mais difícil de alterar.

Se amanhã a aplicação trocar arquivo JSON por banco de dados, vários Controllers podem precisar mudar.

O que é uma dependência?​

Uma dependência é uma classe que outra classe precisa para funcionar.

No exemplo anterior:

private readonly IRepositorio<Caixa> repositorioCaixa;

O CaixaController depende de um IRepositorio<Caixa>.

Ou seja:

para listar, cadastrar, editar ou excluir caixas, o Controller precisa de um objeto capaz de acessar os dados das caixas.

Essa dependência pode ser criada pelo próprio Controller.

Mas também pode ser entregue para ele pronta.

É aí que entra a injeção de dependência.

O que é Injeção de Dependência?​

Injeção de Dependência é uma técnica em que uma classe não cria sozinha os objetos que precisa.

Em vez disso, esses objetos são fornecidos por fora.

No ASP.NET, esse fornecimento normalmente acontece pelo construtor.

Exemplo:

public class CaixaController : Controller
{
    private readonly IRepositorio<Caixa> repositorioCaixa;

    public CaixaController(IRepositorio<Caixa> repositorioCaixa)
    {
        this.repositorioCaixa = repositorioCaixa;
    }
}

Agora o Controller não cria o repositório.

Ele apenas declara:

"para eu funcionar, preciso de um IRepositorio<Caixa>."

Quem decide qual implementação será usada é o ASP.NET.

Como o ASP.NET sabe o que entregar?​

O ASP.NET possui um container de injeção de dependência.

Esse container funciona como um cadastro de serviços.

Nele, registramos quais classes podem ser usadas pela aplicação.

Esse registro normalmente é feito no arquivo Program.cs.

No projeto Clube da Leitura Web 2026, o Program.cs já possui este trecho:

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllersWithViews().AddRazorOptions(options =>
{
    options.ViewLocationFormats.Clear();

    options.ViewLocationFormats.Add("/Modulo{1}/Apresentacao/Views/{0}.cshtml");

    options.ViewLocationFormats.Add("/Compartilhado/Apresentacao/Views/{0}.cshtml");
});

Observe esta parte:

builder.Services.AddControllersWithViews()

O builder.Services representa a coleção de serviços da aplicação.

Quando chamamos AddControllersWithViews, estamos registrando os serviços necessários para o ASP.NET MVC funcionar.

Entre esses serviços estão:

  • Controllers;
  • Views;
  • filtros;
  • validações;
  • estrutura do MVC.

Esse é um primeiro exemplo de injeção de dependência dentro do ASP.NET.

Registrando nossas próprias dependências​

Além dos serviços do MVC, também podemos registrar serviços da nossa aplicação.

No Clube da Leitura Web, já existe uma interface genérica de repositório:

public interface IRepositorio<T> where T : EntidadeBase<T>
{
    void Cadastrar(T entidade);
    bool Editar(string idSelecionado, T entidadeAtualizada);
    bool Excluir(T registro);
    T? SelecionarPorId(string idSelecionado);
    List<T> SelecionarTodos();
    List<T> Filtrar(Predicate<T> filtro);
}

Essa interface descreve o que um repositório deve fazer.

Ela não diz se os dados vêm de:

  • arquivo JSON;
  • banco de dados;
  • memória;
  • API externa.

Isso é importante.

O Controller pode depender da interface.

A implementação concreta fica registrada no Program.cs.

Exemplo com ContextoJson​

O projeto também possui a classe ContextoJson.

Ela representa o contexto usado para salvar e carregar os dados em arquivo JSON.

Um registro possível no Program.cs seria:

builder.Services.AddSingleton<ContextoJson>();

Esse código diz ao ASP.NET:

quando alguém precisar de um ContextoJson, use uma única instância compartilhada pela aplicação.

Depois disso, um repositório pode receber o contexto pelo construtor:

Considerando um módulo de caixas com uma coleção Caixas no contexto, o repositório poderia ficar assim:

public class RepositorioCaixaEmArquivo : RepositorioBaseEmArquivo<Caixa>, IRepositorio<Caixa>
{
    public RepositorioCaixaEmArquivo(ContextoJson contexto) : base(contexto)
    {
    }

    protected override List<Caixa> CarregarRegistros()
    {
        return contexto.Caixas;
    }
}

Nesse exemplo, o repositório também não cria o ContextoJson.

Ele recebe o contexto pronto.

Assim, a injeção de dependência pode acontecer em cadeia:

  1. O Controller recebe o repositório.
  2. O repositório recebe o contexto.
  3. O ASP.NET monta esses objetos na ordem correta.

Registrando repositórios no Program.cs​

Quando o módulo de caixas estiver na aplicação, podemos registrar o repositório assim:

builder.Services.AddScoped<IRepositorio<Caixa>, RepositorioCaixaEmArquivo>();

Esse código diz:

quando alguém pedir um IRepositorio<Caixa>, entregue um RepositorioCaixaEmArquivo.

Depois disso, o Controller pode receber a interface:

public class CaixaController : Controller
{
    private readonly IRepositorio<Caixa> repositorioCaixa;

    public CaixaController(IRepositorio<Caixa> repositorioCaixa)
    {
        this.repositorioCaixa = repositorioCaixa;
    }

    [HttpGet]
    public ActionResult Listar()
    {
        List<Caixa> caixas = repositorioCaixa.SelecionarTodos();

        return View(caixas);
    }
}

Observe a diferença.

O Controller não sabe mais se o repositório usa arquivo, banco de dados ou outro mecanismo.

Ele só sabe que existe um objeto capaz de executar as operações definidas por IRepositorio<Caixa>.

O que significa AddScoped?​

O ASP.NET permite registrar dependências com tempos de vida diferentes.

Os três principais são:

  • AddTransient;
  • AddScoped;
  • AddSingleton.

Cada um controla por quanto tempo uma instância será reutilizada.

AddTransient​

Com AddTransient, uma nova instância é criada sempre que o serviço é solicitado.

Exemplo:

builder.Services.AddTransient<ServicoDeRelatorio>();

Esse formato é útil para serviços leves e sem estado.

Ou seja, serviços que não precisam guardar informações entre chamadas.

AddScoped​

Com AddScoped, uma instância é criada por requisição HTTP.

Exemplo:

builder.Services.AddScoped<IRepositorio<Caixa>, RepositorioCaixaEmArquivo>();

Em aplicações web, esse é um tempo de vida muito comum para repositórios.

Durante uma mesma requisição, a mesma instância pode ser reutilizada.

Na próxima requisição, uma nova instância será criada.

AddSingleton​

Com AddSingleton, uma única instância é criada e reutilizada durante toda a execução da aplicação.

Exemplo:

builder.Services.AddSingleton<ContextoJson>();

Esse formato deve ser usado com cuidado.

Ele combina melhor com objetos compartilhados, configurações e serviços que podem existir uma única vez na aplicação.

Se o objeto guardar dados que mudam muito durante requisições diferentes, é importante avaliar se Singleton realmente é a melhor escolha.

Fluxo completo no ASP.NET​

Com injeção de dependência, o fluxo fica assim:

  1. A aplicação inicia.
  2. O Program.cs registra os serviços no builder.Services.
  3. O ASP.NET monta o servidor com builder.Build().
  4. Uma requisição chega para uma rota.
  5. O ASP.NET identifica o Controller.
  6. O ASP.NET verifica o construtor do Controller.
  7. O container procura as dependências registradas.
  8. O Controller é criado com as dependências prontas.
  9. A action é executada.

Esse processo acontece automaticamente.

Por isso, quando configuramos corretamente o Program.cs, o Controller fica mais limpo.

Erro comum: serviço não registrado​

Se um Controller pedir uma dependência que não foi registrada, o ASP.NET não consegue criar o Controller.

Por exemplo:

public CaixaController(IRepositorio<Caixa> repositorioCaixa)
{
    this.repositorioCaixa = repositorioCaixa;
}

Se não existir um registro como este:

builder.Services.AddScoped<IRepositorio<Caixa>, RepositorioCaixaEmArquivo>();

a aplicação pode gerar um erro parecido com:

InvalidOperationException: Unable to resolve service for type 'IRepositorio<Caixa>'

Esse erro significa:

o ASP.NET sabe que o Controller precisa de um IRepositorio<Caixa>, mas não sabe qual classe deve criar.

A solução é registrar a dependência no Program.cs.

Benefícios da Injeção de Dependência​

Usar injeção de dependência ajuda a criar uma aplicação:

  • mais organizada;
  • mais desacoplada;
  • mais fácil de testar;
  • mais fácil de alterar;
  • mais alinhada ao MVC;
  • mais preparada para crescer.

O Controller deixa de cuidar da criação dos objetos.

Ele passa a cuidar do fluxo da requisição.

O repositório deixa de ser uma escolha fixa dentro do Controller.

Ele passa a ser uma dependência configurada pela aplicação.

Comparando os dois estilos​

Sem injeção de dependência:

public CaixaController()
{
    ContextoJson contexto = new ContextoJson();
    contexto.Carregar();

    repositorioCaixa = new RepositorioCaixaEmArquivo(contexto);
}

Com injeção de dependência:

public CaixaController(IRepositorio<Caixa> repositorioCaixa)
{
    this.repositorioCaixa = repositorioCaixa;
}

O segundo exemplo é menor.

Mas o principal benefício não é apenas escrever menos código.

O principal benefício é separar responsabilidades.

O Controller usa o repositório.

O Program.cs configura qual repositório será usado.

O ASP.NET cria e entrega os objetos necessários.

Conclusão​

Injeção de Dependência é uma forma de entregar para uma classe os objetos que ela precisa para trabalhar.

No ASP.NET, esse mecanismo já faz parte da plataforma.

Registramos os serviços no builder.Services.

Depois, recebemos esses serviços nos construtores dos Controllers e de outras classes.

No Clube da Leitura Web, esse conceito aparece naturalmente quando usamos:

  • Program.cs para registrar serviços;
  • ContextoJson como infraestrutura de persistência;
  • IRepositorio<T> como contrato;
  • repositórios concretos para acessar os dados;
  • Controllers que dependem de interfaces, não de classes concretas.

Com isso, a aplicação fica mais fácil de evoluir.

Se no futuro o Clube da Leitura trocar arquivos JSON por banco de dados, os Controllers podem continuar dependendo de IRepositorio<T>.

A troca principal acontece na configuração e nas implementações registradas.
