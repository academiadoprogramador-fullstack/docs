Camada de Aplicação e Services

Camada de Aplicação e Services​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • ViewModels;
  • validações com ModelState;
  • mensagens com TempData;
  • repositórios;
  • injeção de dependência.

Com isso, já conseguimos criar telas completas para cadastrar, listar, editar e excluir registros.

Porém, conforme a aplicação cresce, surge um problema importante:

o Controller começa a acumular responsabilidades demais.

No começo, isso parece normal.

O Controller recebe a requisição.

Busca dados no repositório.

Valida algumas regras.

Cria objetos do domínio.

Chama o repositório novamente.

Decide qual View ou redirecionamento será retornado.

O problema aparece quando essas regras começam a crescer.

Overview do projeto​

Imagine uma aplicação chamada Clube da Leitura.

Ela possui módulos como:

  • caixas;
  • revistas;
  • amigos;
  • empréstimos.

Uma caixa pode guardar várias revistas.

Uma revista pertence a uma caixa.

Um amigo pode pegar uma revista emprestada.

Com o tempo, as operações deixam de ser apenas "salvar um registro".

Algumas regras começam a aparecer:

  • não cadastrar duas caixas com a mesma etiqueta;
  • não editar uma caixa para uma etiqueta que já existe;
  • não excluir uma caixa que ainda possui revistas;
  • buscar dados de mais de um repositório para validar uma operação;
  • devolver mensagens de erro para a tela quando a operação não puder continuar.

Essas regras fazem parte do funcionamento da aplicação.

Elas não são apenas detalhes visuais da tela.

Por isso, precisamos pensar com cuidado sobre onde colocá-las.

O problema do Controller fazendo tudo​

Em uma primeira versão, o CaixaController poderia concentrar várias responsabilidades.

Ele receberia a requisição.

Leria a ViewModel.

Consultaria repositórios.

Validaria regras.

Criaria entidades.

Salvaria os dados.

Depois decidiria se retorna uma View ou faz um redirecionamento.

Isso funciona.

Mas deixa o Controller cada vez mais difícil de ler.

Um Controller muito grande mistura assuntos diferentes:

  • fluxo da requisição HTTP;
  • validações de tela;
  • regras da aplicação;
  • acesso a dados;
  • criação de entidades;
  • mensagens para o usuário.

Quando tudo fica no mesmo lugar, qualquer mudança passa a parecer mais perigosa.

Alterar uma regra de cadastro pode quebrar uma tela.

Alterar uma mensagem pode misturar com código de repositório.

Reaproveitar uma regra em outra tela se torna mais difícil.

Exemplo de regra dentro do Controller​

Antes de usar uma camada de aplicação, uma Action de cadastro poderia validar a etiqueta da caixa diretamente:

List<Caixa> caixas = repositorioCaixa.SelecionarTodos();

foreach (Caixa c in caixas)
{
    if (string.Equals(c.Etiqueta, cadastrarVm.Etiqueta, StringComparison.OrdinalIgnoreCase))
    {
        ModelState.AddModelError(
            nameof(cadastrarVm.Etiqueta),
            "Já existe uma caixa com esta etiqueta."
        );
    }
}

Esse código funciona.

Mas observe o que o Controller passou a conhecer:

  • como buscar todas as caixas;
  • como comparar etiquetas;
  • qual campo da tela deve receber o erro;
  • qual mensagem deve aparecer;
  • em qual momento a regra deve ser executada.

Se outra Action precisar validar uma etiqueta parecida, podemos acabar repetindo a mesma lógica.

Esse é um sinal comum de que a aplicação precisa de uma nova camada.

O que é a Camada de Aplicação?​

A Camada de Aplicação é a parte do sistema que organiza os casos de uso da aplicação.

Um caso de uso é uma operação que o usuário ou o sistema precisa executar.

Exemplos:

  • cadastrar uma caixa;
  • editar uma caixa;
  • excluir uma caixa;
  • listar caixas;
  • buscar os detalhes de uma caixa.

A camada de aplicação fica entre:

  • a camada de apresentação, onde ficam Controllers, Views e ViewModels;
  • a camada de domínio, onde ficam entidades e regras centrais do negócio;
  • a camada de infraestrutura, onde ficam repositórios, arquivos, banco de dados ou APIs externas.

Em uma aplicação MVC, ela ajuda o Controller a ficar mais focado no fluxo web.

O Controller continua recebendo a requisição.

Mas ele delega a execução da operação para uma classe de serviço.

O que é um Service?​

Um Service é uma classe que representa uma operação ou um conjunto de operações da aplicação.

No módulo de caixas, podemos criar um ServicoCaixa.

Ele pode ficar responsável por operações como:

  • cadastrar caixas;
  • editar caixas;
  • excluir caixas;
  • listar caixas;
  • buscar detalhes de uma caixa.

Exemplo:

public class ServicoCaixa
{
    private readonly IRepositorioCaixa repositorioCaixa;
    private readonly IRepositorioRevista repositorioRevista;

    public ServicoCaixa(
        IRepositorioCaixa repositorioCaixa,
        IRepositorioRevista repositorioRevista
    )
    {
        this.repositorioCaixa = repositorioCaixa;
        this.repositorioRevista = repositorioRevista;
    }
}

Observe que o Service também recebe dependências pelo construtor.

Isso segue o mesmo princípio que já vimos com injeção de dependência.

O Service não cria os repositórios manualmente.

Ele apenas declara que precisa deles.

O papel do Service​

O Service não deve cuidar de HTML.

Ele não deve decidir qual View será exibida.

Ele não deve depender de ModelState, TempData ou Controller.

O papel dele é executar a operação da aplicação.

Por exemplo:

  • validar se a operação pode acontecer;
  • consultar repositórios;
  • criar ou alterar entidades;
  • chamar métodos de persistência;
  • devolver um resultado para o Controller.

Com isso, o Controller fica mais simples.

Ele conversa com a web.

O Service executa a operação.

Cadastrando uma caixa com Service​

Com um Service, a regra de cadastro pode ficar dentro do ServicoCaixa:

public Result Cadastrar(CadastrarCaixaDto dto)
{
    if (ExisteCaixaComEtiqueta(dto.Etiqueta))
        return Falha("Etiqueta", "Já existe uma caixa com esta etiqueta.");

    Caixa novaCaixa = new Caixa(
        dto.Etiqueta,
        dto.Cor,
        dto.DiasDeEmprestimo
    );

    repositorioCaixa.Cadastrar(novaCaixa);

    return Result.Ok();
}

Agora a regra está em um lugar mais adequado.

O Service verifica se a etiqueta já existe.

Se existir, ele retorna uma falha.

Se não existir, ele cria a entidade Caixa e salva no repositório.

O Controller não precisa mais saber como essa regra é feita.

Ele só precisa chamar o serviço e interpretar o resultado.

O que é um DTO?​

DTO significa Data Transfer Object.

Em português, podemos entender como:

um objeto simples usado para transportar dados entre camadas.

Exemplo:

public record CadastrarCaixaDto(
    string Etiqueta,
    string Cor,
    int DiasDeEmprestimo
);

Esse DTO representa os dados necessários para cadastrar uma caixa.

Ele não é uma ViewModel.

Ele também não é uma entidade do domínio.

Ele é um objeto usado para levar os dados do Controller até o Service.

ViewModel, DTO e Entidade​

Nesse ponto, é importante separar três tipos de objetos:

TipoOnde costuma ser usadoResponsabilidade
ViewModelApresentaçãoRepresentar os dados da tela
DTOAplicaçãoTransportar dados entre camadas
EntidadeDomínioRepresentar conceitos do negócio

No cadastro de caixa, o fluxo fica assim:

  1. A View envia os dados do formulário.
  2. O Controller recebe uma CadastrarCaixaViewModel.
  3. O Controller cria um CadastrarCaixaDto.
  4. O Service recebe o DTO.
  5. O Service cria uma entidade Caixa.
  6. O repositório salva a entidade.

Cada objeto tem um papel.

Isso evita misturar detalhes da tela com regras da aplicação.

Controller depois do Service​

Depois da criação do Service, o Controller passa a depender dele:

public class CaixaController : Controller
{
    private readonly ServicoCaixa servicoCaixa;

    public CaixaController(ServicoCaixa servicoCaixa)
    {
        this.servicoCaixa = servicoCaixa;
    }
}

Essa mudança é pequena, mas importante.

Antes, o Controller precisava de repositórios.

Depois, ele passa a precisar de um serviço de aplicação.

Isso deixa o Controller mais próximo da sua responsabilidade principal:

receber requisições HTTP e retornar respostas HTTP.

Action de cadastro usando Service​

A Action de cadastro pode ficar mais focada no fluxo da tela:

[HttpPost]
public ActionResult Cadastrar(CadastrarCaixaViewModel cadastrarVm)
{
    if (!ModelState.IsValid)
        return View(cadastrarVm);

    CadastrarCaixaDto dto = new CadastrarCaixaDto(
        cadastrarVm.Etiqueta,
        cadastrarVm.Cor,
        cadastrarVm.DiasDeEmprestimo
    );

    Result resultado = servicoCaixa.Cadastrar(dto);

    if (resultado.IsFailed)
    {
        foreach (IError erro in resultado.Errors)
        {
            string campo =
                erro.Metadata["Campo"] is string ? erro.Metadata["Campo"].ToString()! : string.Empty;

            ModelState.AddModelError(campo, erro.Message);
        }

        return View(cadastrarVm);
    }

    return RedirectToAction(nameof(Listar));
}

Agora a Action faz três coisas principais:

  1. valida os dados básicos da tela;
  2. chama o Service;
  3. transforma falhas em mensagens para a View.

A regra de etiqueta duplicada não está mais espalhada dentro da Action.

Ela está no ServicoCaixa.

Retornando resultados com FluentResults​

Uma forma prática de representar sucesso ou falha em um Service é usar a biblioteca FluentResults.

Ela permite que um método retorne o resultado da operação de forma explícita.

Exemplo:

return Result.Ok();

Esse retorno representa uma operação bem-sucedida.

Exemplo de falha:

return Result.Fail("Caixa não encontrada.");

Esse retorno representa uma operação que não pôde ser concluída.

No ServicoCaixa, isso pode aparecer em métodos como Editar, Excluir e SelecionarPorId.

Assim, o Service não precisa lançar exceções para erros esperados do fluxo da aplicação.

Ele pode devolver um resultado para o Controller decidir como apresentar isso ao usuário.

Erros associados a campos​

Em alguns casos, a falha precisa apontar para um campo específico do formulário.

Por exemplo, quando a etiqueta já existe.

No ServicoCaixa, podemos criar um método auxiliar:

private static Result Falha(string campo, string mensagem)
{
    IError erro = new Error(mensagem).WithMetadata("Campo", campo);

    return Result.Fail(erro);
}

Esse método cria uma falha com uma mensagem e com o nome do campo.

Depois, o Controller lê esse campo e adiciona o erro ao ModelState.

Com isso, a View consegue exibir a mensagem no lugar certo.

Excluindo uma caixa com regra de aplicação​

Uma caixa não deve ser excluída se ainda possuir revistas relacionadas.

Essa regra envolve mais de um repositório:

  • o repositório de caixas;
  • o repositório de revistas.

Se essa regra ficar no Controller, a Action de exclusão precisa conhecer detalhes demais:

List<Revista> revistas = repositorioRevista.SelecionarTodos();

foreach (Revista r in revistas)
{
    if (string.Equals(r.Caixa.Id, excluirVm.Id))
    {
        TempData["MensagemErro"] = "Esta caixa não pode ser excluída pois está relacionada a uma revista.";

        return RedirectToAction(nameof(Listar));
    }
}

Essa regra é importante para a aplicação.

Mas essa decisão não é uma responsabilidade da tela.

No Service, ela pode ficar assim:

public Result Excluir(string id)
{
    Caixa? caixa = repositorioCaixa.SelecionarPorId(id);

    if (caixa == null)
        return Result.Fail("Caixa não encontrada.");

    List<Revista> revistas = repositorioRevista.SelecionarTodos();

    foreach (Revista r in revistas)
    {
        if (string.Equals(r.Caixa.Id, id))
            return Result.Fail("Esta caixa não pode ser excluída pois está relacionada a uma revista.");
    }

    repositorioCaixa.Excluir(id);

    return Result.Ok();
}

Agora o Controller só precisa chamar o serviço:

Result resultado = servicoCaixa.Excluir(excluirVm.Id);

if (resultado.IsFailed)
    TempData["MensagemErro"] = resultado.Errors.First().Message;

return RedirectToAction(nameof(Listar));

O Controller continua cuidando do TempData e do redirecionamento.

O Service cuida da regra da aplicação.

Registrando o Service no Program.cs​

Como o ServicoCaixa é uma dependência do Controller, ele precisa ser registrado no container de injeção de dependência.

No Program.cs, podemos registrar:

builder.Services.AddScoped<ServicoCaixa>();

Esse registro diz ao ASP.NET:

quando alguém precisar de um ServicoCaixa, crie uma instância durante a requisição atual.

Como o ServicoCaixa também depende de repositórios, eles também precisam estar registrados:

builder.Services.AddScoped<IRepositorioCaixa, RepositorioCaixaEmArquivo>();
builder.Services.AddScoped<IRepositorioRevista, RepositorioRevistaEmArquivo>();

Assim, o ASP.NET consegue montar a cadeia completa:

  1. O CaixaController precisa de ServicoCaixa.
  2. O ServicoCaixa precisa de IRepositorioCaixa e IRepositorioRevista.
  3. O container encontra as implementações registradas.
  4. O ASP.NET cria os objetos e entrega tudo pronto.

O que deve ficar no Controller?​

O Controller ainda tem responsabilidades importantes.

Ele deve cuidar de assuntos ligados à web e à tela.

Exemplos:

  • receber ViewModels;
  • validar ModelState;
  • chamar Services;
  • montar ViewModels para retornar à View;
  • usar TempData para mensagens temporárias;
  • decidir entre View e RedirectToAction;
  • retornar respostas HTTP.

O Controller não precisa saber todos os detalhes de uma regra de aplicação.

Ele precisa saber qual operação chamar.

O que deve ficar no Service?​

O Service deve cuidar do fluxo da operação.

Exemplos:

  • cadastrar uma entidade;
  • editar uma entidade;
  • excluir uma entidade;
  • impedir operações inválidas;
  • consultar repositórios necessários;
  • retornar sucesso ou falha;
  • coordenar regras que envolvem mais de um repositório.

No caso do ServicoCaixa, isso fica evidente na exclusão.

Para excluir uma caixa, não basta acessar o repositório de caixas.

Também é necessário verificar o repositório de revistas.

Esse tipo de coordenação combina bem com a camada de aplicação.

O que não deve ficar no Service?​

O Service não deve virar uma cópia do Controller.

Evite colocar nele:

  • ModelState;
  • TempData;
  • ViewBag;
  • ActionResult;
  • nomes de Views;
  • código HTML;
  • detalhes de formulário.

Esses elementos pertencem à camada de apresentação.

O Service deve conseguir executar a operação sem depender de uma tela específica.

Isso facilita reaproveitar a mesma regra em outros lugares no futuro.

Por exemplo:

  • uma API;
  • uma tela diferente;
  • um teste automatizado;
  • uma rotina interna da aplicação.

Benefícios da Camada de Aplicação​

Usar Services ajuda a criar uma aplicação:

  • mais organizada;
  • mais fácil de alterar;
  • mais fácil de testar;
  • com Controllers menores;
  • com regras menos duplicadas;
  • com responsabilidades mais claras;
  • mais preparada para crescer.

O principal benefício não é apenas diminuir o tamanho do Controller.

O principal benefício é separar motivos de mudança.

Se a regra de cadastro mudar, alteramos o Service.

Se a forma de exibir erro mudar, alteramos o Controller ou a View.

Se a persistência mudar, alteramos a infraestrutura.

Cada parte do sistema fica com uma responsabilidade mais definida.

Fluxo completo com camada de aplicação​

Com a camada de aplicação, o fluxo de cadastro fica assim:

  1. O usuário envia o formulário.
  2. O Controller recebe a ViewModel.
  3. O Controller verifica o ModelState.
  4. O Controller cria um DTO.
  5. O Controller chama o Service.
  6. O Service valida as regras da operação.
  7. O Service cria ou altera entidades.
  8. O Service chama o repositório.
  9. O Service retorna sucesso ou falha.
  10. O Controller decide se retorna a View ou redireciona.

Esse fluxo parece maior quando escrito em etapas.

Mas no código ele deixa cada classe mais simples.

Cada uma faz uma parte menor do trabalho.

Comparando antes e depois​

Antes da camada de aplicação:

Controller
+ recebe requisição
+ valida tela
+ busca dados
+ aplica regras
+ cria entidades
+ salva no repositório
+ cria mensagens
+ retorna View ou redirecionamento

Depois da camada de aplicação:

Controller
+ recebe requisição
+ valida tela
+ chama Service
+ cria mensagens
+ retorna View ou redirecionamento

Service
+ aplica regras
+ busca dados
+ cria entidades
+ chama repositórios
+ retorna sucesso ou falha

Essa divisão deixa o projeto mais fácil de entender.

O aluno que abrir o Controller entende o fluxo da página.

O aluno que abrir o Service entende as operações da aplicação.

Conclusão​

A camada de aplicação aparece quando o Controller começa a ter responsabilidades demais.

Em vez de deixar regras importantes espalhadas nas Actions, criamos Services para organizar os casos de uso da aplicação.

No exemplo do Clube da Leitura, o ServicoCaixa cuida das operações de caixa.

O Controller continua responsável pela comunicação com a tela.

Os repositórios continuam responsáveis pelo acesso aos dados.

Essa separação deixa o código mais claro e prepara o projeto para crescer com menos repetição e menos acoplamento.
