Extension Methods

Extension Methods no C#​

Até agora, já utilizamos:

  • Controllers;
  • Views;
  • ViewModels;
  • DTOs;
  • Services;
  • ModelState;
  • TempData;
  • resultados com FluentResults.

Com isso, já conseguimos separar melhor as responsabilidades da aplicação.

O Controller recebe a requisição.

O Service executa a operação.

O resultado volta para o Controller, que decide se retorna uma View ou redireciona o usuário.

Porém, conforme esse fluxo se repete em vários módulos, aparece um novo problema:

algumas pequenas operações começam a se repetir em muitos Controllers.

Por exemplo:

  • pegar a primeira mensagem de erro de um Result;
  • transformar erros do Service em erros do ModelState;
  • adicionar uma mensagem de erro no TempData.

Essas operações são pequenas.

Mas, quando aparecem em várias Actions, deixam o Controller mais poluído.

Para resolver esse tipo de repetição, o C# possui um recurso chamado Extension Methods.

O problema da repetição no Controller​

Imagine uma Action de cadastro usando um Service.

Quando o Service retorna uma falha, o Controller precisa adicionar os erros ao ModelState:

if (resultado.IsFailed)
{
    foreach (IError erro in resultado.Errors)
    {
        string campo = string.Empty;

        if (erro.Metadata.TryGetValue("Campo", out object? campoErro))
            campo = campoErro?.ToString() ?? string.Empty;

        ModelState.AddModelError(campo, erro.Message);
    }

    return View(cadastrarVm);
}

Esse código funciona.

Mas ele mistura muitos detalhes dentro da Action:

  • como percorrer os erros;
  • como ler o metadado "Campo";
  • como tratar campo nulo;
  • como chamar AddModelError.

A Action deveria ficar mais focada no fluxo da tela.

Ela deveria dizer:

se o resultado falhou, adicione os erros no ModelState e retorne a View.

O detalhe de como os erros são adicionados pode ficar em outro lugar.

O que é um Extension Method?​

Um Extension Method é um método que parece ter sido adicionado a uma classe já existente.

Em português, podemos entender como:

um método de extensão permite criar novos comportamentos para um tipo sem alterar a classe original.

Por exemplo, o ModelStateDictionary é uma classe do ASP.NET.

Ela já possui métodos próprios, como:

ModelState.AddModelError("Etiqueta", "Já existe uma caixa com esta etiqueta.");

Mas podemos criar um método novo para facilitar o nosso uso:

ModelState.AdicionarErros(resultado);

Esse método não existe originalmente no ASP.NET.

Ele é criado por nós como um Extension Method.

Como criar um Extension Method​

Para criar um Extension Method em C#, precisamos de três coisas:

  • uma classe static;
  • um método static;
  • o modificador this no primeiro parâmetro do método.

Exemplo:

public static class ResultExtensions
{
    public static string ObterMensagemErro(this ResultBase result)
    {
        return result.Errors.First().Message;
    }
}

Observe esta parte:

this ResultBase result

O this indica que o método será usado como se pertencesse ao tipo ResultBase.

Por isso, em vez de chamar assim:

ResultExtensions.ObterMensagemErro(resultado);

podemos chamar assim:

resultado.ObterMensagemErro();

O segundo formato fica mais natural de ler.

Ele parece um método do próprio objeto resultado.

Exemplo com FluentResults​

No Clube da Leitura Web, os Services podem retornar objetos Result e Result<T> da biblioteca FluentResults.

Quando uma operação falha, é comum precisar pegar a primeira mensagem de erro.

Sem Extension Method, o Controller poderia fazer:

TempData["MensagemErro"] = resultado.Errors.First().Message;

Esse código funciona.

Mas ele obriga o Controller a conhecer a estrutura interna do resultado.

Podemos melhorar isso criando uma extensão:

namespace ClubeDaLeituraWeb.WebApp.Compartilhado.Aplicacao;

public static class ResultExtensions
{
    public static string ObterMensagemErro(this ResultBase result)
    {
        return result.Errors.First().Message;
    }
}

Agora o Controller pode escrever:

TempData["MensagemErro"] = resultado.ObterMensagemErro();

O código ficou mais próximo da intenção.

Em vez de mostrar como a mensagem é buscada, ele mostra o que queremos fazer.

Entendendo o ResultBase​

No exemplo, o método recebe um ResultBase:

public static string ObterMensagemErro(this ResultBase result)

Isso é útil porque ResultBase é uma base comum para resultados do FluentResults.

Assim, o mesmo Extension Method pode ser usado com:

  • Result;
  • Result<T>.

Por exemplo:

Result resultadoCadastro = servicoCaixa.Cadastrar(dto);

Result<DetalhesCaixaDto> resultadoBusca = servicoCaixa.SelecionarPorId(id);

Os dois podem usar:

resultadoCadastro.ObterMensagemErro();
resultadoBusca.ObterMensagemErro();

Isso evita criar um método separado para cada tipo de resultado.

Extension Method para ModelState​

Outro ponto repetitivo aparece quando precisamos enviar erros do Service para a View.

O Service pode retornar erros com metadados:

return Result.Fail(
    new Error("Já existe uma caixa com esta etiqueta.")
        .WithMetadata("Campo", "Etiqueta")
);

Esse erro possui duas informações importantes:

  • a mensagem que será exibida;
  • o campo da tela relacionado ao erro.

No Controller, queremos transformar isso em erro do ModelState.

Para isso, podemos criar uma extensão para ModelStateDictionary:

namespace ClubeDaLeituraWeb.WebApp.Compartilhado.Apresentacao;

public static class ModelStateExtensions
{
    public static void AdicionarErros(this ModelStateDictionary modelState, ResultBase result)
    {
        foreach (IError erro in result.Errors)
        {
            string campo = ObterCampo(erro);

            modelState.AddModelError(campo, erro.Message);
        }
    }

    private static string ObterCampo(IError erro)
    {
        if (erro.Metadata.TryGetValue("Campo", out object? campo))
            return campo?.ToString() ?? string.Empty;

        return string.Empty;
    }
}

Agora a Action pode ficar assim:

if (resultado.IsFailed)
{
    ModelState.AdicionarErros(resultado);

    return View(cadastrarVm);
}

Esse código é menor.

Mas o principal benefício não é apenas diminuir linhas.

O principal benefício é dar um nome claro para uma operação que se repete.

Por que ObterCampo é private?​

Dentro de ModelStateExtensions, existe este método:

private static string ObterCampo(IError erro)

Ele é private porque só é usado pela própria classe de extensão.

O Controller não precisa conhecer esse detalhe.

O Controller só precisa chamar:

ModelState.AdicionarErros(resultado);

Isso mantém a parte mais específica escondida.

Se no futuro a forma de buscar o campo mudar, alteramos apenas a extensão.

As Actions continuam iguais.

Extension Method para TempData​

Também podemos criar uma extensão para o TempData.

Sem Extension Method, usamos:

TempData["MensagemErro"] = mensagemErro;

Com uma extensão, podemos deixar a intenção mais explícita:

namespace ClubeDaLeituraWeb.WebApp.Compartilhado.Apresentacao;

public static class TempDataExtensions
{
    public static void AdicionarErro(this ITempDataDictionary tempData, string mensagemErro)
    {
        tempData["MensagemErro"] = mensagemErro;
    }
}

Agora o Controller pode escrever:

TempData.AdicionarErro("Caixa não encontrada.");

Ou, usando junto com a extensão de Result:

TempData.AdicionarErro(resultado.ObterMensagemErro());

Esse tipo de método ajuda a padronizar as chaves usadas no TempData.

Em vez de espalhar "MensagemErro" por vários Controllers, deixamos essa chave centralizada em um único lugar.

Usando as extensões no Controller​

Depois de criar os Extension Methods, o CaixaController pode ficar mais limpo.

Observe principalmente as chamadas:

  • ModelState.AdicionarErros(resultado);
  • TempData.AdicionarErro(...);
  • resultado.ObterMensagemErro().
public class CaixaController(ServicoCaixa servicoCaixa, IMapper mapeador) : Controller
{
    [HttpGet]
    public ActionResult Listar()
    {
        List<ListarCaixasDto> dtos = servicoCaixa.SelecionarTodos();

        List<ListarCaixasViewModel> listarVms = mapeador.Map<List<ListarCaixasViewModel>>(dtos);

        return View(listarVms);
    }

    [HttpGet]
    public ActionResult Cadastrar()
    {
        CadastrarCaixaViewModel cadastrarVm = new CadastrarCaixaViewModel(
            string.Empty,
            string.Empty,
            7
        );

        return View(cadastrarVm);
    }

    [HttpPost]
    public ActionResult Cadastrar(CadastrarCaixaViewModel cadastrarVm)
    {
        if (!ModelState.IsValid)
            return View(cadastrarVm);

        CadastrarCaixaDto dto = mapeador.Map<CadastrarCaixaDto>(cadastrarVm);

        Result resultado = servicoCaixa.Cadastrar(dto);

        if (resultado.IsFailed)
        {
            ModelState.AdicionarErros(resultado);

            return View(cadastrarVm);
        }

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Editar(string id)
    {
        Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

        if (resultado.IsFailed)
        {
            TempData.AdicionarErro(resultado.ObterMensagemErro());

            return RedirectToAction(nameof(Listar));
        }

        EditarCaixaViewModel editarVm =
            mapeador.Map<EditarCaixaViewModel>(resultado.Value);

        return View(editarVm);
    }

    [HttpPost]
    public ActionResult Editar(EditarCaixaViewModel editarVm)
    {
        if (!ModelState.IsValid)
            return View(editarVm);

        EditarCaixaDto dto = mapeador.Map<EditarCaixaDto>(editarVm);

        Result resultado = servicoCaixa.Editar(dto);

        if (resultado.IsFailed)
        {
            ModelState.AdicionarErros(resultado);

            return View(editarVm);
        }

        return RedirectToAction(nameof(Listar));
    }

    [HttpGet]
    public ActionResult Excluir(string id)
    {
        Result<DetalhesCaixaDto> resultado = servicoCaixa.SelecionarPorId(id);

        if (resultado.IsFailed)
        {
            TempData.AdicionarErro(resultado.ObterMensagemErro());

            return RedirectToAction(nameof(Listar));
        }

        ExcluirCaixaViewModel excluirVm =
            mapeador.Map<ExcluirCaixaViewModel>(resultado.Value);

        return View(excluirVm);
    }

    [HttpPost]
    public ActionResult Excluir(ExcluirCaixaViewModel excluirVm)
    {
        Result resultado = servicoCaixa.Excluir(excluirVm.Id);

        if (resultado.IsFailed)
            TempData.AdicionarErro(resultado.ObterMensagemErro());

        return RedirectToAction(nameof(Listar));
    }
}

O Controller ainda faz o papel dele.

Ele recebe ViewModels.

Chama o Service.

Mapeia DTOs e ViewModels.

Decide entre View e RedirectToAction.

Mas os detalhes repetitivos de erro ficaram em métodos reutilizáveis.

O Service continua cuidando das regras​

Os Extension Methods não substituem a camada de aplicação.

Eles apenas ajudam o Controller a lidar melhor com o resultado retornado pelo Service.

As regras continuam no ServicoCaixa:

public class ServicoCaixa(IRepositorioCaixa repositorioCaixa, IRepositorioRevista repositorioRevista)
{
    public Result Cadastrar(CadastrarCaixaDto dto)
    {
        if (ExisteCaixaComEtiqueta(dto.Etiqueta))
        {
            return Result.Fail(
                new Error("Já existe uma caixa com esta etiqueta.")
                    .WithMetadata("Campo", "Etiqueta")
            );
        }

        Caixa novaCaixa = new Caixa(dto.Etiqueta, dto.Cor, dto.DiasDeEmprestimo);

        repositorioCaixa.Cadastrar(novaCaixa);

        return Result.Ok();
    }

    public Result Editar(EditarCaixaDto dto)
    {
        if (ExisteCaixaComEtiqueta(dto.Etiqueta, dto.Id))
        {
            return Result.Fail(
                new Error("Já existe uma caixa com esta etiqueta.")
                    .WithMetadata("Campo", "Etiqueta")
            );
        }

        Caixa caixaAtualizada = new Caixa(dto.Etiqueta, dto.Cor, dto.DiasDeEmprestimo);

        bool conseguiuEditar = repositorioCaixa.Editar(dto.Id, caixaAtualizada);

        if (!conseguiuEditar)
            return Result.Fail("Caixa não encontrada.");

        return Result.Ok();
    }

    public Result Excluir(string id)
    {
        Caixa? caixa = repositorioCaixa.SelecionarPorId(id);

        if (caixa == null)
            return Result.Fail("Caixa não encontrada.");

        bool possuiRevistasVinculadas = repositorioRevista
            .SelecionarTodos()
            .Any(r => r.Caixa.Id == caixa.Id);

        if (possuiRevistasVinculadas)
            return Result.Fail("Não é possível excluir uma caixa com revistas vinculadas.");

        repositorioCaixa.Excluir(caixa);

        return Result.Ok();
    }

    public Result<DetalhesCaixaDto> SelecionarPorId(string id)
    {
        Caixa? caixa = repositorioCaixa.SelecionarPorId(id);

        if (caixa == null)
            return Result.Fail("Caixa não encontrada.");

        return new DetalhesCaixaDto(id, caixa.Etiqueta, caixa.Cor, caixa.DiasDeEmprestimo);
    }

    public List<ListarCaixasDto> SelecionarTodos()
    {
        return repositorioCaixa.SelecionarTodos()
            .Select(c => new ListarCaixasDto(c.Id, c.Etiqueta, c.Cor, c.DiasDeEmprestimo))
            .ToList();
    }

    private bool ExisteCaixaComEtiqueta(string etiqueta, string? idIgnorado = null)
    {
        return repositorioCaixa.SelecionarTodos().Any(c =>
            c.Id != idIgnorado &&
            string.Equals(c.Etiqueta.Trim(), etiqueta?.Trim(), StringComparison.OrdinalIgnoreCase)
        );
    }
}

Observe a separação:

  • o Service decide se a operação pode acontecer;
  • o Service retorna Result.Ok() ou Result.Fail(...);
  • o Controller interpreta o resultado;
  • os Extension Methods ajudam o Controller a converter erros em mensagens para a tela.

Cada parte continua com uma responsabilidade clara.

Onde colocar Extension Methods​

Extension Methods normalmente ficam em classes separadas.

No Clube da Leitura Web, faz sentido separar por contexto.

Por exemplo:

Compartilhado
+ Aplicacao
  + ResultExtensions.cs
+ Apresentacao
  + ModelStateExtensions.cs
  + TempDataExtensions.cs

Essa organização ajuda a indicar a intenção de cada extensão.

ResultExtensions está ligada ao resultado da camada de aplicação.

ModelStateExtensions e TempDataExtensions estão ligadas à apresentação, porque dependem de recursos do ASP.NET MVC.

Importando o namespace​

Para usar um Extension Method, o arquivo precisa enxergar o namespace onde ele foi declarado.

Por exemplo:

using ClubeDaLeituraWeb.WebApp.Compartilhado.Aplicacao;
using ClubeDaLeituraWeb.WebApp.Compartilhado.Apresentacao;

Se o namespace não estiver importado, o método pode não aparecer.

Nesse caso, o código abaixo pode gerar erro:

resultado.ObterMensagemErro();

O método existe.

Mas o compilador ainda não sabe onde encontrá-lo.

Adicionar o using correto resolve o problema.

Quando usar Extension Methods​

Use Extension Methods quando você quer adicionar um comportamento simples a um tipo que já existe.

Eles combinam bem com:

  • operações pequenas e repetidas;
  • código de apoio para frameworks;
  • leitura mais natural de objetos;
  • padronização de chaves e mensagens;
  • conversões simples entre estruturas.

No nosso exemplo, eles ajudam a trabalhar com:

  • ResultBase;
  • ModelStateDictionary;
  • ITempDataDictionary.

Quando evitar Extension Methods​

Extension Methods não devem ser usados para esconder regras grandes.

Evite criar extensões que:

  • executam regras importantes da aplicação;
  • acessam banco de dados;
  • alteram muitos objetos ao mesmo tempo;
  • deixam o fluxo difícil de entender;
  • viram uma forma de esconder código complexo.

Se a lógica representa um caso de uso, provavelmente ela pertence a um Service.

Se a lógica representa uma pequena operação de apoio, um Extension Method pode fazer sentido.

No exemplo das caixas:

  • validar etiqueta duplicada pertence ao ServicoCaixa;
  • transformar erros em ModelState pode ficar em ModelStateExtensions;
  • guardar mensagem no TempData pode ficar em TempDataExtensions.

Benefícios dos Extension Methods​

Usar Extension Methods ajuda a:

  • reduzir repetição em Controllers;
  • melhorar a leitura das Actions;
  • dar nomes claros para operações comuns;
  • centralizar pequenos padrões da aplicação;
  • reaproveitar código sem alterar classes do ASP.NET;
  • manter Controllers mais focados no fluxo da requisição.

O ganho aparece principalmente quando vários Controllers precisam fazer a mesma coisa.

Em vez de repetir blocos de código parecidos, criamos uma extensão com um nome claro.

Depois usamos esse método nos lugares necessários.

Conclusão​

Extension Methods permitem adicionar métodos a tipos já existentes sem modificar a classe original.

No C#, eles são criados em classes static, com métodos static, usando this no primeiro parâmetro.

No Clube da Leitura Web, eles ajudam a simplificar o trabalho do Controller com resultados, erros e mensagens temporárias.

Com eles, podemos escrever:

ModelState.AdicionarErros(resultado);
TempData.AdicionarErro(resultado.ObterMensagemErro());

Essas chamadas deixam a intenção mais clara.

O Controller continua cuidando do fluxo web.

O Service continua cuidando das regras da aplicação.

E os Extension Methods ficam responsáveis por pequenas operações repetitivas que tornam o código mais limpo e padronizado.
